## 一、先分清三种东西

| 形态 | 定义 | 典型 |
|------|------|------|
| **确定性工作流** | 步骤固定，分支人定 | 代码 if/else、Airflow、自研状态机 |
| **Agent 循环** | 模型动态选下一步/工具 | ReAct 自研循环 |
| **图编排框架** | 节点+边+状态，可环可人机 | **LangGraph**、部分商业编排 |

LangGraph：在 LangChain 生态上，用**有状态图**编排 LLM 应用，支持循环、分支、持久化、中断恢复（HITL）。

## 二、对比维度（面试表）

| 维度 | LangGraph | 自研工作流/状态机 | 纯 ReAct 自研 |
|------|-----------|-------------------|---------------|
| 可控性 | 图结构清晰，边可约束 | 最强，全代码 | 弱，靠 prompt |
| 灵活性 | 中高，可动态路由节点 | 改代码发版 | 最高 |
| 可观测 | 生态+自埋点 | 完全自定义 | 需自建 |
| 持久化/恢复 | 检查点能力 | 自建 | 难 |
| HITL | 一等公民 interrupt | 自建审批节点 | 难 |
| 学习/依赖 | 绑定生态与版本 | 无包袱 | 轻 |
| 团队 | 要懂框架抽象 | 要能设计状态机 | 易写难控 |
| 适用 | 复杂 Agent、多步带环 | 稳定业务 SOP | Demo/简单工具调用 |

## 三、选型建议（P0 结论）

| 场景 | 建议 |
|------|------|
| 固定审批/ETL/简单 RAG 链 | **自研工作流或纯代码管道** |
| 多步工具、需循环重试、人机确认、状态恢复 | **LangGraph（或同类图编排）** |
| 只要单轮工具调用 | Function Calling + 薄封装即可 |
| 强合规、少依赖、核心链路 | 自研状态机 + 明确节点 |
| 快速验证 Agent 产品 | LangGraph/开源框架加速 |

**原则：** 能确定的步骤不要交给模型乱走；模型只放在需要智能决策的节点。

## 四、自研工作流最小设计

```text
State（上下文）
→ Node 函数（纯/带 IO）
→ Transition（条件边）
→ 持久化 state + run_id
→ 超时/重试/补偿
→ 轨迹日志
```

坑：状态膨胀、节点无幂等、没有版本、难回放。

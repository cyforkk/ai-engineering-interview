## 1. 索引为什么快？InnoDB 用什么结构？

- 索引减少扫描行数，避免全表扫描。
- InnoDB 常用 **B+ 树**：矮胖、适合磁盘、支持范围查询。
- **主键索引（聚簇）**：叶子节点是**整行数据**。
- **二级索引**：叶子存**主键值**，常需**回表**再查聚簇。
- **覆盖索引**：查询列都在二级索引中，无需回表。

---

## 2. 联合索引与最左前缀？

- 索引 `(a,b,c)` 按 a、b、c 排序。
- 能用：`a` / `a,b` / `a,b,c`。
- 通常不能很好用：只有 `b` 或只有 `c`（无 a）。
- 设计时把区分度高、常出现在条件里的列放左边（结合查询模式）。

---

## 3. 索引失效常见情况？

- 对索引列使用函数 / 运算。
- `LIKE '%xx'` 左模糊。
- 类型隐式转换。
- 违反最左前缀。
- 优化器认为全表扫描更便宜。
- **以 EXPLAIN 为准**（看 key、type、rows、Extra）。

---

## 4. 事务 ACID？隔离级别？三种读问题？

- **ACID**：原子性、一致性、隔离性、持久性。
- **读问题**
  - 脏读：读到未提交数据。
  - 不可重复读：同一事务两次读同一行结果不同（被改）。
  - 幻读：范围读行数变化（插入/删除）。
- InnoDB **默认 RR（可重复读）**。

---

## 5. MVCC 是什么？

- 多版本并发控制：行有版本链（undo）。
- **快照读**：普通 `SELECT`，按 ReadView 看版本，一般不加锁。
- **当前读**：`SELECT ... FOR UPDATE` / 更新删除等，读最新并加锁。

---

## 6. 慢 SQL 怎么排查？

1. 慢查询日志 / 监控定位 SQL。  
2. `EXPLAIN` 看执行计划。  
3. 加索引 / 改写 SQL / 避免 `SELECT *`。  
4. 验证耗时与计划。  
- 深分页：`LIMIT 很大偏移` 差 → 用 `id > last` 或延迟关联。

---

## 7. redo / undo / binlog？

- **redo**：崩溃恢复，保证已提交不丢（WAL）。
- **undo**：回滚 + MVCC 旧版本。
- **binlog**：主从复制、归档恢复。

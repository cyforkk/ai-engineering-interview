## 1. 索引：为啥能快？

### 先记一句

没索引 = 一本书**没有目录**，找人名只能从头翻到尾。  
有索引 = 有目录，很快翻到那一页。

### InnoDB 里常见是 B+ 树

你可以想成：**多层目录**。  
第一层指到大章节，再指到小节，最后指到那一行。树不深，磁盘读几次就能定位。

### 主键索引和普通索引差在哪？

- **主键（聚簇）**：目录的最底层**直接就是整行数据**（人带行李住一起）  
- **普通二级索引**：最底层往往只记**主键号**，还要再拿主键去取整行 → 叫**回表**（先查到房号，再回房间拿行李）

如果查询要的字段，二级索引里全有了，就不用回表 → **覆盖索引**，更快。

### 联合索引 (a,b,c)：为啥强调「最左」？

目录是按 a，再 b，再 c 排序的。  
你不告诉 a，只说 b，目录没法从左边开始翻——所以常常用不上。  
条件里有 a，或 a+b，才好用。

### 哪些情况索引容易「失效」？

用人话：

- 对索引列做了函数：`WHERE YEAR(date)=2024`（先把列加工了，目录对不上）  
- `LIKE '%张'`（前边模糊，目录排序帮不上）  
- 类型乱转（字符串列你传了数字等）  

最终听 **EXPLAIN**：看它到底用没用索引。

## 2. 事务：一伙操作要成对成功

### 生活例子

转账：A 减钱、B 加钱。只成功一半就乱了。  
事务 = **要么一起成功，要么一起当没发生**。

### 三个吓人词

| 词 | 人话 |
|----|------|
| 脏读 | 读到别人**还没提交**的修改（人家可能马上撤销） |
| 不可重复读 | 同一事务里读两次同一行，中间被人改了，两次不一样 |
| 幻读 | 范围查询，中间有人插入/删除，**行数变了**像见鬼 |

MySQL InnoDB 默认隔离级别多是 **可重复读（RR）**。

### MVCC 是啥？（能说人话就行）

一行数据可以有**多个版本**（改之前、改之后）。  
普通查询像拍了张**快照**：在你这个事务里，尽量看到一致的一版，还不用老加锁堵着别人。  
`SELECT ... FOR UPDATE` 这种是**要看最新的并锁住**，和快照读不一样。

## 3. 慢 SQL 怎么办？

四步，按顺序：

1. 先找到慢的那条（慢查询日志/监控）  
2. `EXPLAIN` 看有没有走索引、扫了多少行  
3. 加合适索引或改写法  
4. 再测一次，确认真的快了  

**深分页**为啥痛：`LIMIT 1000000,10` 像翻到第 100 万页再取 10 条，前面白干很多。  
更好：`WHERE id > 上次最后id LIMIT 10` 这种「从上次接着走」。

## 二、索引：为什么是 B+ 树

### 问题本质

磁盘 IO 一次很贵。要在千万行里找一行，必须**尽量少次 IO** 定位。树越矮、每层分叉越多，IO 越少。

### 为何偏 B+ 而不是二叉树/哈希

- 二叉树太高，IO 次数多  
- 哈希点查快但**范围查询、排序**弱  
- B+：内部节点主要存键与指针，叶子存数据（或主键）并顺序串联 → **点查与范围**都合适，贴合 SQL 的 `WHERE`/`ORDER BY`/`BETWEEN`

### 聚簇与回表的所以然

InnoDB 主键索引叶子就是**整行**（聚簇）——主键点查最快。  
二级索引叶子只存**主键值**，否则每个二级索引都复制整行，空间与维护成本爆炸。代价是：二级索引命中后往往还要再查聚簇 → **回表**。  
**覆盖索引**让查询列都在二级索引里，省掉回表，所以「联合索引列顺序」和「SELECT 列表」都值得设计。

### 最左前缀为什么

联合索引 (a,b,c) 在 B+ 里按 a、b、c 排序。没有 a 的条件时，索引无法从左边有序二分切入（一般情况），故「最左」。这是排序结构的必然，不是玄学规定。

### 失效的本质

优化器选「成本更低」的计划。对列套函数、类型转换、左模糊会让索引**无法按序定位**，成本变高，于是可能全表扫描。以 `EXPLAIN` 为准。

## 三、事务与隔离：为什么需要

并发事务交叉读写，会出现脏读、不可重复读、幻读。隔离级别是**正确性与并发度的旋钮**：级别越高锁/多版本成本越大，并发可能越差。

InnoDB 默认 **RR**，并配合 MVCC 与间隙锁等机制，在常见业务下平衡体验。

### MVCC 为什么能提高读并发

若每次读都加锁，读写互相堵。  
多版本：写生成新版本，读按 ReadView 看自己该看的旧/新版本 → **快照读**多数情况无需与写死锁式互斥。  
当前读（`FOR UPDATE`）必须看最新并阻止冲突写，所以要加锁。

## 四、日志三分：redo / undo / binlog

| 日志 | 解决什么问题 |
|------|----------------|
| redo | 崩溃后如何恢复已提交数据（WAL，先日志后落页） |
| undo | 回滚；MVCC 旧版本来源 |
| binlog | 复制、审计、时间点恢复（Server 层） |

两阶段提交思想保证 redo 与 binlog **同事务维度一致**，否则主从和崩溃恢复会对不齐。

---

## 五、慢 SQL 方法论为什么是这四步

1. **先量化**（慢日志）——避免凭感觉优化  
2. **EXPLAIN**——看是否用错计划  
3. **改索引/SQL**——对准根因  
4. **验证**——防「越优越慢」  

深分页慢：因为数据库仍可能扫描并丢弃大量行；`id > last` 把工作变成范围扫描起点推进。

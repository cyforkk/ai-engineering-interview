## 二、索引（P0）

### 白话

没索引 = 一本书没有目录，找人名只能从头翻到尾（**全表扫描**）。  
InnoDB 索引是 **B+ 树**：多层目录，叶子层串联，范围查询友好。

- **主键索引（聚簇）**：叶子节点就是 **整行数据**。  
- **二级索引**：叶子存 **主键值**；查完二级往往还要拿主键再回聚簇查一次 → **回表**。  
- **覆盖索引**：查询列都在二级索引里，不必回表，更快。  
- **联合索引 (a,b,c)**：从左往右用，叫 **最左前缀**；只写 `b=?` 通常用不上这个索引。

**常见失效：** 对索引列包函数、`LIKE '%xx'`、隐式类型转换、优化器判断全表更便宜。最终以 `EXPLAIN` 为准。

### 面试怎么答（约 2 分钟）

> InnoDB 用 B+ 树做索引。主键是聚簇索引，叶子就是行数据；二级索引叶子存主键，需要时回表。  
> 联合索引遵守最左前缀；能覆盖查询就少回表。  
> 建索引看区分度和查询条件，不是越多越好，写放大和空间都要代价。  
> 我会用 EXPLAIN 看 type、key、rows、Extra，确认有没有用上索引、是否 Using filesort/temporary。

### 追问

| 问 | 答 |
|----|-----|
| 为何不用二叉树？ | 太高，磁盘 IO 次数多；B+ 更矮更胖 |
| 索引越多越好？ | 否，占空间拖慢写 |
| 主键为何推荐自增？ | 顺序插入减少页分裂（也有业务用 UUID 的权衡） |

## 三、事务、隔离级别、MVCC（P0）

### 白话

**事务**：一组操作要么全成功要么全回滚。ACID：原子、一致、隔离、持久。

三种读现象：

| 现象 | 白话 |
|------|------|
| 脏读 | 读到别人 **未提交** 的修改 |
| 不可重复读 | 同一事务两次读同一行，中间被别人 **改了** |
| 幻读 | 范围查询行数变了（别人插入/删除） |

隔离级别从低到高大致：读未提交 → 读已提交（RC）→ 可重复读（RR）→ 串行化。  
InnoDB **默认 RR**。

**MVCC（人话）：** 一行有多个版本（undo 链）。普通 SELECT 是 **快照读**，按事务的 ReadView 决定能看见哪个版本，一般 **不加锁**。  
`SELECT ... FOR UPDATE` 是 **当前读**，读最新并加锁。

### 面试怎么答（约 2 分钟）

> 事务保证一组写操作的原子和持久。隔离级别权衡并发和异常读。  
> MySQL InnoDB 默认可重复读。  
> 实现上靠 MVCC：行版本链加 ReadView，快照读不加锁提高并发；当前读才加锁。  
> 脏读/不可重复读/幻读我会各举一句例子说明差异。RR 下间隙锁等机制用来缓解幻读，细节可再展开。

**完整稿：** [MySQL面渣 · 事务](./MySQL面渣级口述.md)

## 四、锁与日志（P1）

### 白话要点

- InnoDB **行锁** 本质加在索引上；条件没走索引可能锁更多行甚至退化。  
- 死锁：互相等锁，InnoDB 检测后 **回滚成本小的一方**。  
- **redo log**：崩溃恢复，保证已提交不丢（WAL 思想）。  
- **undo log**：回滚 + MVCC 版本。  
- **binlog**：主从复制、数据恢复。  
- 事务提交涉及 redo 与 binlog 的一致（两阶段提交思想）。

### 面试怎么答

> 行锁提高并发，但 SQL 要走对索引。  
> redo 保持久，undo 保回滚和多版本，binlog 保复制。  
> 面试若问崩溃恢复，我会从 WAL 和 redo 讲起。

## 五、慢 SQL 排查（P0 工程向）

### 可照着说的四步

```text
1. 慢查询日志 / 监控找到慢 SQL
2. EXPLAIN 看是否全表、用的哪个索引、扫描行数
3. 改：加合适索引、改写 SQL、避免选*、处理深分页
4. 在预发验证耗时与执行计划，再上线
```

深分页：`LIMIT 1000000,10` 很痛 → 用 `id > last_id` 或延迟关联。

### 面试怎么答

> 先量化慢在哪，再 EXPLAIN，再改索引或 SQL，最后用数据验证。  
> 不会一上来就「加个索引」而不看执行计划。

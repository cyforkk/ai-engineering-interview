## P0 长口述

### 1. 什么是 Agent？和 Chat、RAG、工作流区别

**口述：**

> Agent 是目标驱动的 LLM 系统：不只生成一段话，还能在循环里**规划下一步、调用工具、观察结果、再决策**，直到任务完成或触发停止条件。  
> 经典拆法是：大模型 + 规划 + 记忆 + 工具。  
>
> 和普通 Chat 比：Chat 主要是多轮生成；Agent 有行动和外部副作用，比如查库、调 API。  
> 和 RAG 比：RAG 是检索增强生成的一种能力，常作为 Agent 的一个工具；纯 RAG 链步骤通常固定，Agent 会动态决定要不要检索、检索几次。  
> 和工作流比：工作流步骤人预先定好，可控可测；Agent 步骤由模型动态选，灵活但更不可控、更贵。  
> 和微调比：微调改模型行为风格，不替代实时知识和行动。  
>
> 工程上我们会限制步数、权限和人机审批，避免「无限自主」。

---

### 2. 讲讲 ReAct，如何防止死循环

**口述：**

> ReAct 是 Reason + Act：模型先 Thought 推理，再选 Action 调工具，拿到 Observation，再 Thought，循环，直到 Final Answer。  
> 优点是边做边看，适合工具场景，轨迹也可解释。  
> 缺点是容易短视、步数飘、重复调同一工具、工具选错还死磕。  
>
> 防死循环我一般做这些：  
> 硬限制 max_steps、总超时、总 token/费用预算；  
> 检测重复的 Action+参数；  
> 工具连续失败熔断转人工或降级；  
> 中间强制摘要压缩上下文；  
> 关键写操作 HITL 人工确认；  
> 复杂长任务改成 Plan-and-Execute 或「能工作流就工作流」。  
> 线上把平均步数、重复率做成监控。

---

### 3. Function Calling / 工具调用全流程

**口述：**

> 我们把工具用 JSON Schema 描述清楚：名字、干什么、参数类型和必填。  
> 模型不是随便输出自然语言去「假装调用」，而是输出结构化的 tool call。  
> 运行时解析参数，在服务端真实执行，把结果作为 Observation 塞回对话。  
> 模型再决定继续调工具还是给最终答案。  
>
> 工具设计要点：描述要让模型选得准；参数约束要严；错误返回要结构化方便重试；写操作和读操作权限分离；执行层再做鉴权，**绝不信任模型参数**。  
> 长结果要摘要或存指针，避免撑爆上下文。

---

### 4. 如何保证 Agent 安全（乱调工具）

**口述：**

> 模型可能被提示注入或幻觉去调危险工具。  
> 手段：工具白名单；最小权限账号；高危操作二次确认；参数校验与 allowlist；代码执行进沙箱；系统提示与不可信检索内容隔离；全链路审计。  
> 生产默认不开「任意 shell」这类能力。

## P1 长口述

### 5. 记忆怎么设计

**口述：**

> 分短时和长时。  
> 短时是当前会话消息和本轮工具轨迹，用滑动窗口，太长就摘要。  
> 长时是用户偏好、历史事实，用向量库或结构化画像，需要时检索注入。  
> 原则是控制噪声：不是什么都塞进 prompt，而是相关的才拿出来。  
> 错误记忆要能覆盖删除，带来源和 TTL。

---

### 6. Agent 里 RAG 放哪

**口述：**

> 两种常见：一是固定前置检索再进 Agent；二是把 search_knowledge_base 做成工具，让 ReAct 自己决定何时检索。  
> 第二种更灵活，但可能多步、成本高。  
> RAG 本身要做好分块、混合检索、重排、引用；无依据要拒答，减少幻觉当事实。  
> 评估看忠实度、答案相关性、以及任务是否真完成。

---

### 7. Plan-and-Execute 和 ReAct 怎么选

**口述：**

> ReAct 适合短链路、要探索试工具的任务。  
> Plan-and-Execute 先出全局计划再逐步执行，适合步骤多、依赖清楚的任务，计划过时可 replan。  
> 复杂系统往往混合：上层计划，子任务 ReAct；或核心路径工作流，边缘用 Agent。  
> 可控性和成本经常比「看起来更智能」更重要。

---

### 8. 如何评估一个 Agent

**口述：**

> 分层指标：工具是否选对、参数是否对、任务完成率、步数、耗时、费用、人工接管率。  
> 有金标集做回归；轨迹回放看卡在哪一步；在线看用户反馈和业务转化。  
> 只看「模型答得像不像人」不够，Agent 要看任务是否闭环成功。

## P2 长口述

### 9. 多 Agent 何时用

**口述：**

> 角色天然分工、需要交叉校验、或可并行时再上。  
> 默认先单 Agent + 清晰工具。多 Agent 要有 Supervisor、共享状态、轮次上限，否则互相踢皮球、费用爆炸。  
> 死锁靠明确输出契约和总控裁决。

### 10. MCP 是什么

**口述：**

> MCP 是模型上下文协议，让工具和资源以统一方式被发现和调用，偏生态互联。  
> Function Calling 是模型 API 侧的工具调用格式。  
> 一个偏协议生态，一个偏单次调用形态，可以一起出现在架构里。

---

## 系统设计口述模板

> 我先定目标和边界，明确不能自动做什么。  
> 架构上接入鉴权，中间用状态图或 Agent 循环编排，下面挂工具和 RAG，旁路记忆和观测。  
> 规划默认 ReAct 限步，高风险 HITL。  
> 安全鉴权注入审计。  
> 评估用离线集+在线完成率。  
> 成本上小模型路由，能缓存缓存。  
> 演进从工作流逐步放开 Agent 节点，而不是一上来全自动。

```text
记忆：窗口+摘要+向量
评估：完成率/工具正确率/费用
```

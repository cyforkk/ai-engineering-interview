## 项目名称：电商促销活动中台（库存 + 下单）

| 字段 | 填写示例 |
|------|----------|
| 业务一句话 | 为大促活动提供统一库存扣减与下单入口，支撑多 SKU 限购，避免超卖 |
| 你的角色 | 后端开发，负责库存扣减链路、缓存一致性、下单接口与部分监控 |
| 规模数据 | 大促峰值约 **2k QPS**（估算）；SKU 约 5 万；单机 4C8G × 8 实例 |
| 技术栈 | Java 17、Spring Boot、Redis、MySQL、RocketMQ、Gateway 限流 |
| 最大难点 1 | 热点 SKU 超卖 + Redis/DB 双写一致 |
| 方案与取舍 | Redis Lua 预扣扛并发 + MQ 异步落单 + DB 条件更新兜底；**不用**强一致分布式事务（成本高、热点锁死）；缺点是最终一致窗口内要做对账与回补 |
| 结果 | 大促超卖工单 **0**；下单 P99 从 ~800ms 到 ~200ms；库存对账差异 &lt;0.01% 可自动回补 |
| 复盘 | 热点 key 应更早做本地缓存+分片；对账任务最初是全表扫，应改增量 binlog |

### 可挂靠的高频场景

| 场景 | 挂靠 | 项目里对应什么 | 2 分钟口述要点 |
|------|:----:|----------------|----------------|
| 高并发 / 限流 | ☑ | 网关 + 接口限流 | ① 网关按用户/IP 限流 ② 服务内令牌桶保护库存 ③ 超限快速失败不拖垮 DB |
| 缓存三连 / 双写 | ☑ | 库存缓存 | ① 穿透：空值短 TTL ② 击穿：互斥/逻辑过期 ③ 更新：先更 DB 再删缓存，接受短暂不一致 |
| 分布式锁 | ☑ | 冷门 SKU 补货任务 | ① Redis SET NX + 过期 ② 持有者 token 防误解 ③ 业务更依赖 Lua 原子扣，锁只用于离线任务 |
| MQ 不丢不重 | ☑ | 下单异步 | ① 生产者确认 ② 消费者幂等（订单号唯一键）③ 积压扩消费者+跳过毒丸 |
| 慢 SQL / 索引 | ☑ | 订单查询 | ① 活动维度联合索引 ② 避免大 offset ③ 大促只读从库 |
| 线程池 / 异步 | ☑ | 通知与对账 | ① 自建池禁 Executors ② 队列有界 ③ 拒绝策略可观察 |
| 秒杀 / 库存 | ☑ | 核心 | ① 漏斗限流 ② Redis 预扣 ③ MQ ④ DB `stock=stock-1 where stock>0` |
| 幂等 | ☑ | 下单/支付回调 | ① 客户端单号 ② 唯一索引 ③ 状态机防重放 |
| JVM 排查 | ☐ | 未直接挂 | — |

### STAR 口述稿（约 3 分钟）

- **S：** 公司大促前，活动库存多次出现超卖与下单超时，业务要求「可卖尽卖但不能超卖」。  
- **T：** 我负责改造库存扣减与下单主路径，目标是峰值可抗、超卖为 0、可观测。  
- **A：** ① 设计漏斗：网关限流 → 服务校验限购 → Redis Lua 原子扣 → 成功才发 MQ 落单。② DB 用条件更新做最终校验，失败回补 Redis。③ 订单号幂等；对账任务修差异。④ 看板：扣减失败率、MQ 堆积、对账差异。  
- **R：** 大促 0 超卖；P99 明显下降；复盘文档沉淀为活动中台标准链路。  

### 面试官可能追问（示例答）

1. **为什么不先锁 DB？**  
   热点行锁会把数据库打满；预扣把竞争前移到 Redis，DB 只处理成功候选单。  
2. **Lua 失败或 Redis 挂了怎么办？**  
   失败快速返回；Redis 故障可降级「只读展示+排队」或切只 DB（会降吞吐），不能静默超卖。  
3. **重复扣减？**  
   扣减与订单绑定 requestId；幂等表；Lua 里带业务键去重。

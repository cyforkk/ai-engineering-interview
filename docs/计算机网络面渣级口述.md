## P0 长口述

### 1. TCP 和 UDP 区别

**口述：**

> TCP 面向连接、可靠、有序、有流量和拥塞控制，适合 HTTP、文件传输、MySQL 连接这种不能乱不能丢的。  
> UDP 无连接、不保证可靠有序，开销小延迟低，适合 DNS 查询、直播、部分游戏。  
> 可靠不是协议高级就一定好，要看场景：视频丢一帧可以接受，银行转账不行。

---

### 2. 三次握手（为什么不是两次）

**口述：**

> 第一次客户端发 SYN，说我想连，带初始序号。  
> 第二次服务端回 SYN+ACK，既确认客户端，也带上自己的序号。  
> 第三次客户端回 ACK，确认服务端的序号。  
> 之后进入建立成功，可以传数据。  
>
> 为什么不是两次？两次的话，如果一个旧的延迟 SYN 到了服务端，服务端以为新连接建立并分配资源，但客户端并没有真正要连，造成资源浪费；三次让客户端再次确认，避免历史连接误建立。  
> 也确认了双方的发送和接收能力都正常。

---

### 3. 四次挥手与 TIME_WAIT

**口述：**

> TCP 是全双工，每个方向单独关闭，所以往往四次。  
> 主动方发 FIN；被动方 ACK 表示收到；被动方数据处理完再发 FIN；主动方再 ACK。  
> 主动方最后进入 **TIME_WAIT**，等 2MSL。作用：一是保证最后的 ACK 丢了对方会重传 FIN，还能再回；二是让旧连接数据包在网络中消失，避免混进新连接。  
> 高并发短连接会导致大量 TIME_WAIT，需要调内核参数或改连接复用策略，但不能粗暴以为 TIME_WAIT 全是坏事。

---

### 4. HTTP 和 HTTPS

**口述：**

> HTTP 明文传输，中间人能窃听篡改。  
> HTTPS 是 HTTP over TLS：先用证书证明服务器身份，用非对称加密交换会话密钥，再用对称加密传数据，又快又安全。  
> 还能一定程度防篡改。代价是握手多一点 CPU 和延迟，现在硬件和会话复用下可接受。  
> 证书由 CA 签发，浏览器校验证书链和域名。

---

### 5. GET 和 POST、状态码

**口述：**

> 语义上 GET 取资源，应安全幂等；POST 提交，可能改数据。  
> GET 参数常在 URL，可缓存；POST 在 body。实际很多人混用，但面试按语义答。  
> 状态码：2xx 成功；3xx 重定向；4xx 客户端问题如 401 未认证、403 无权限、404 没有；5xx 服务端 500、网关 502、过载 503、超时 504。

## P1 长口述

### 6. 从输入 URL 到页面显示

**口述：**

> 解析 URL → DNS 解析成 IP → 建立 TCP（HTTPS 再 TLS 握手）→ 发 HTTP 请求 → 服务器处理返回响应 → 浏览器解析 HTML 建 DOM、CSSOM，加载 JS，渲染绘制。  
> 过程中有缓存、CDN、HTTP2 多路复用等优化点。  
> 这题考你能否串起网络各层，不必背每一毫秒细节。

---

### 7. HTTP/1.1 和 HTTP/2

**口述：**

> 1.1 有 keep-alive 长连接，但队头阻塞，同连接请求响应仍受顺序影响。  
> 2 用二进制分帧、多路复用，一个连接并行多个流，还有头部压缩。  
> 3 基于 QUIC/UDP，进一步改善弱网和连接迁移。说到 2 的多路复用即可加分。

---

### 8. 粘包半包

**口述：**

> TCP 是字节流，没有消息边界，所以应用层会出现粘包半包。  
> 解决：定长、分隔符、或最常用**长度头+body**。Netty 里有现成解码器。  
> 这是传输层可靠字节流和应用层消息语义的差别问题。

---

### 9. Cookie 和 Session

**口述：**

> HTTP 无状态，用 Cookie 把小数据存在客户端，每次请求带上。  
> Session 把会话存在服务端，客户端只持有 session id。  
> Token/JWT 把状态放在令牌里，利于无状态扩展，要注意过期和注销。

## 考前骨架

```text
TCP 可靠有连；UDP 快无连
握手三步防旧 SYN；挥手四次+TIME_WAIT
HTTPS = HTTP+TLS 证书+混合加密
URL：DNS→TCP→TLS→HTTP→渲染
```

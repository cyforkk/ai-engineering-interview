## 一、答题框架（先说这个）

`	ext
澄清需求 → 容量粗算 → 架构分层 → 关键路径 → 优化 → 兜底监控
`

面试官要的是权衡能力，不是背标准架构图。

---

## 二、澄清我会问什么

- 功能范围：读写路径、核心实体  
- 非功能：QPS 峰值、数据量、延迟、一致性强弱、可用性  
- 是否允许最终一致、主要端是 App 还是内部  

粗算：DAU × 人均行为 × 峰均比 → 读写 QPS；存储按条数×大小×副本。

---

## 三、通用架构积木

接入 CDN/网关 → 无状态服务集群 → Redis 缓存 → MySQL → MQ 异步 → 对象存储。  
手段：限流熔断降级、幂等、分库分表、多级缓存、可观测。

---

## 四、秒杀 90 秒浓缩

漏斗：前端防抖 → 网关限流 → 库存 Redis 原子预扣 → MQ 异步下单 → DB 条件更新兜底。  
防超卖：原子扣 + WHERE stock>0；防重：幂等单号；热点：分段库存、本地缓存。  
失败：回补库存、对账。

---

## 五、其他锚点

短链：发号 Base62 + 缓存跳转。  
Feed：推/拉/推拉结合。  
支付：状态机 + 幂等。  
细节见各场景口述与完整卷。

---

## 六、收尾

每方案说清为什么、缺点、兜底；尽量映射自己项目的限流、缓存、异步经验。

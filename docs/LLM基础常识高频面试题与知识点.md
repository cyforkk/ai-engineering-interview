## 先建立正确心态

大模型很会说话，但：

- 它不是实时搜索引擎  
- 它会**一本正经胡说**  
- 它按 **token** 计长度和钱，不是按「字数感觉」

工程师要管：**成本、长度、稳定性、胡话风险**。

---

## 1. Token 和上下文窗口

### 生活例子

Token ≈ 模型吃饭的「小块饼干」。  
一句话会被掰成很多小块；中文常常掰得更碎 → **更贵、更容易塞满**。

上下文窗口 ≈ **餐盘大小**。  
系统设定、历史聊天、检索来的资料、用户问题、还要留空给回答——都要装进盘子。  
盘子满了：不是模型突然变笨，是**装不下或被截断**。

### 工程上怎么办？

历史太长就**做摘要**；检索别一股脑塞 50 段；控制每段长度。

---

## 2. 温度：稳还是活

### 生活例子

- 温度低：像考试填空，尽量选最稳的答案  
- 温度高：像头脑风暴，更跳脱，也更容易跑偏  

客服、抽字段、调工具参数 → **偏低**。  
写小说可以略高。

---

## 3. 幻觉：为啥压不灭只能管

### 先记一句

模型在猜「下一个词更像什么」，不是在查公司数据库。  
所以会说得流畅但**是错的**。

### 怎么办？（面试常答）

1. 先检索公司知识再答（RAG），尽量带引用  
2. 资料不相关就说「不知道」  
3. 关键数字做校验  
4. 用固定考题集做评测，防止越改越差  

不要承诺「某某模型永不胡说」。

---

## 4. 预训练 / SFT / 对齐

| 阶段 | 人话 |
|------|------|
| 预训练 | 读海量文本，学会说话和常识 |
| SFT | 用问答例子教会「听人话」 |
| 对齐 | 更有用、更安全、更符合人类偏好 |

公司知识天天变 → 优先 **RAG**；只要固定语气/格式 → 才考虑微调。

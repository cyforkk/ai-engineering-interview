## 一、本专题要搞定什么

AI 工程岗开场题，不考你推 Transformer 公式，考：

1. **Token / 上下文窗口** 是否懂计费与长度  
2. **幻觉** 是否当成工程风险而不是玄学  
3. 温度等参数什么时候调  
4. 预训练 / SFT / 偏好对齐 各干什么  
5. 云 API vs 私有化怎么选型

## 二、Token 与上下文窗口（P0）

### 白话

模型不直接「按字」思考，而是把文本切成 **token**（常见是子词）。  
中文往往比英文 **更吃 token**（同样一段话 token 数更多）。  
API 通常按 **输入 token + 输出 token** 计费。  

**上下文窗口**：一次请求里模型最多能「看见」的长度，包括系统提示、历史对话、RAG 检索片段、用户问题，还要 **预留输出空间**。  
满了不是越拼越好，而是要：摘要历史、少而精检索、截断旧轮。

### 面试怎么答（约 1 分钟）

> Token 是模型的基本文本单位，也是计费和长度限制单位。中文通常更耗 token。  
> 上下文窗口是一次能塞下的上限，系统提示、历史、检索结果、问题都占窗口。  
> 超窗时我会做历史摘要、控制检索条数和单段长度，而不是无限拼接。  
> 工程上还会监控每请求 token，做成本与延迟预算。

### 追问

| 问 | 答 |
|----|-----|
| 和字符数关系？ | 不是 1:1，和分词器有关 |
| TTFT？ | Time To First Token，首 token 延迟，流式体验关键 |

## 三、温度与采样（P0）

### 白话

生成下一个 token 时带随机性。  

- **温度低（如 0～0.3）**：更确定，适合抽字段、分类、客服标准答、要稳定的工具参数。  
- **温度高**：更发散，适合 brainstorm，但更不稳定、更易跑飞。  

事实性、结构化输出：**偏低**。创意文案：可以略高。

### 面试怎么答

> 温度控制随机性。生产里事实与结构化任务我倾向低温；需要多样性再升高。  
> 还会配合 top_p 等，但面试先把温度场景说清即可。

## 四、幻觉（P0，必会）

### 白话

**幻觉**：说得流畅、语气很像真的，但内容是错的或捏造的。  
根因：模型本质是 **根据概率续写**，不是实时查权威数据库。它「会说话」≠「知道真相」。

**工程应对（不要只说换模型）：**

1. **RAG grounding**：让回答基于检索到的材料并尽量引用  
2. 低相关 **拒答** 或「材料不足」  
3. 结构化校验、规则、二次检查  
4. **评测集** 回归，而不是手测两句  

### 面试怎么答（约 1 分钟）

> 幻觉是大模型生成式特性，不是偶发 bug。  
> 业务上我会用检索增强把答案锚定到可引用材料，相关度低就拒答，关键字段做校验，并用金标集持续评测。  
> 不会承诺「某个模型永不胡说」，而是把风险压到业务可接受。

## 五、预训练 / SFT / RLHF（P0）

| 阶段 | 白话 | 解决什么 |
|------|------|----------|
| 预训练 | 海量文本学语言与知识 | 「会说话、有知识」 |
| SFT | 指令-回答数据微调 | 「会按人的要求答」 |
| RLHF/偏好对齐 | 人类/AI 偏好优化 | 更有用、更安全、更听话 |

业务落地：

- 知识常变、要可引用 → **优先 RAG**  
- 语气/格式/领域任务形态 → Prompt 或 SFT  
- 安全与偏好 → 对齐 + 策略层  

### 面试怎么答

> 预训练打底，SFT 教会遵循指令，偏好对齐让表现更符合人类期望与安全。  
> 业务知识更新快时我优先 RAG；只有稳定风格或特殊格式成本合适才考虑微调。

## 六、云 API vs 私有化（P1）

| | 云 API | 私有化/本地 |
|--|--------|-------------|
| 优点 | 快上线、省运维、模型新 | 数据可控、可定制 |
| 代价 | 数据出域、单价、依赖供应商 | GPU、运维、效果迭代 |

常见：**网关统一**，敏感走私有，一般走云；都要超时、限流、降级、审计。

### 面试怎么答

> 看数据敏感度、调用量、延迟与团队运维能力。  
> 很多团队混合部署，用统一 LLM 网关屏蔽差异。

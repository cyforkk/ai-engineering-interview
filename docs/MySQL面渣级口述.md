## P0 长口述

### 1. 为什么索引用 B+ 树

**口述：**

> InnoDB 用 B+ 树做索引，主要因为磁盘 IO 贵，要尽量矮胖、一次读更多键。  
> 和 B 树比：B+ 非叶子节点不存行数据只存键和指针，扇出更大，树更矮；数据都在叶子，叶子之间有链表，**范围查询、排序很友好**。  
> 和 Hash 比：Hash 等值快，但不支持范围、排序，也不适合最左前缀这种有序需求。  
> 所以 OLTP 这种既有等值又有范围的场景，B+ 是更好的默认选择。

---

### 2. 聚簇索引、二级索引、回表、覆盖索引

**口述：**

> InnoDB 表数据按**聚簇索引**组织，一般是主键，叶子节点就是整行数据。  
> **二级索引**叶子存的是主键值，不是整行。所以用二级索引查非索引列时，要再拿主键回聚簇索引查一次，叫**回表**。  
> 如果查询的列都在二级索引里，就不用回表，叫**覆盖索引**，Extra 里常见 Using index，通常更快。  
> 没有主键时 InnoDB 会生成隐藏 row_id 做聚簇。  
> 所以主键最好稳定、自增，避免随机 UUID 导致页分裂和二级索引膨胀。

---

### 3. 最左前缀和索引失效

**口述：**

> 联合索引 (a,b,c) 是按 a、b、c 顺序排序的，必须从左边开始用，这叫最左前缀。  
> where a=?、a=? and b=?、a b c 都能较好用上；只 b 或只 c 通常用不上这个联合索引。  
> a=? and c=? 可能只用到 a，c 无法在索引里连续用。  
>
> 失效常见情况：对索引列做函数或运算、隐式类型转换比如字符串列和数字比、LIKE '%xx' 前导模糊、违反最左、OR 一侧没索引、优化器觉得全表更划算。  
> 是不是失效最终以 **explain** 为准，别死记绝对规则。

---

### 4. 事务隔离级别和三种读问题

**口述：**

> 事务有 ACID：原子、一致、隔离、持久。隔离靠锁和 MVCC 等机制保证。  
> 四种隔离级别从低到高：读未提交、读已提交、可重复读、串行化。  
> 脏读是读到别人没提交的数据；不可重复读是同一事务两次读同一行，中间被别人改了；幻读是范围读的行数变了，中间有人插入删除。  
> MySQL InnoDB **默认可重复读 RR**。  
> 读未提交三种问题都可能有；RC 解决脏读；RR 解决不可重复读，对幻读靠 MVCC 和间隙锁等进一步控制；串行化最严，性能最差。

---

### 5. MVCC 原理

**口述：**

> MVCC 是多版本并发控制，让读写尽量不阻塞。  
> 每行背后有版本信息，更新会通过 undo 形成版本链。普通 SELECT 是**快照读**，根据 ReadView 判断哪个版本对自己可见，不用加锁。  
> SELECT FOR UPDATE 或更新删除是**当前读**，读最新并加锁。  
> RC 下每条语句可能生成新的 ReadView，所以同事务两次读可能看到别的事务已提交的更新。  
> RR 下 ReadView 在事务中更稳定（通常首次读建立后复用），所以可重复读。  
> 幻读在快照读下被 MVCC 缓解，当前读范围场景还要靠间隙锁、临键锁。

---

### 6. InnoDB 和 MyISAM

**口述：**

> InnoDB 支持事务、行级锁、外键、崩溃恢复，用聚簇索引，是默认引擎。  
> MyISAM 是表锁、不支持事务，以前适合只读，现在基本淘汰。  
> 选 InnoDB 就是为了并发和数据安全。

## P1 长口述

### 7. 行锁、间隙锁、死锁

**口述：**

> InnoDB 行锁是加在索引上的。如果更新条件没走索引，可能锁得更狠，甚至接近表级效果，所以索引对锁粒度也很重要。  
> RR 下为了防幻读，有间隙锁锁住索引间隙，阻止别人插入；临键锁是记录锁加间隙锁。  
> 死锁是事务互相等锁。InnoDB 会检测并回滚代价小的一方。  
> 业务上减少死锁：短事务、相同顺序访问行、合适索引减小锁范围、避免大事务扫很多行。

---

### 8. redo、undo、binlog

**口述：**

> redo log 是 InnoDB 的，保证持久性，先写日志再慢慢刷脏页，崩溃用 redo 恢复，属于 WAL 思想。  
> undo log 用于回滚，也支撑 MVCC 版本链。  
> binlog 是 Server 层的，逻辑日志，用于主从复制和归档恢复。  
> 两者都要时，用**两阶段提交**保证 redo 和 binlog 一致，避免主从和崩溃恢复数据不一致。

---

### 9. 慢 SQL 怎么优化

**口述：**

> 第一步用慢查询日志或监控把 SQL 抓出来。  
> 第二步 explain 看 type、key、rows、Extra：有没有全表扫、用没用上索引、是否 Using filesort/temporary。  
> 第三步改：加合适索引、避免索引失效、覆盖索引、改写 SQL、避免 select *、控制返回行数。  
> 深分页 limit 1000000,10 很慢，改成基于上次最大 id 的翻页，或延迟关联先查主键再 join。  
> 第四步验证执行时间和 explain 变化。再不行考虑缓存、汇总表、读写分离。  
> 加索引不是越多越好，写放大和维护成本要考虑。

---

### 10. 主从复制和延迟

**口述：**

> 主库写 binlog，从库拉过去存 relay log 再回放。用于读写分离、备份、高可用。  
> 延迟常见因为从库回放慢或大事务。  
> 读写分离要处理读到旧数据：关键读走主、会话级读主、延迟监控与降级。

## 场景口述

### 超卖

> 不能先查库存再更新，有并发窗口。用事务里 `update ... set stock=stock-1 where id=? and stock>=1`，看影响行数；或乐观锁版本号。热点行再考虑分段库存或队列串行化。

### 线上偶发死锁

> 打开死锁日志或 show engine innodb status，看两个事务的 SQL 和锁。通常是索引缺失或更新顺序不一致。补索引、统一顺序、拆小事务。

## 考前骨架

```text
索引：B+、最左、回表/覆盖、失效→explain
事务：RR 默认、脏/不可重复/幻
MVCC：版本链+ReadView；快照读/当前读
日志：redo 持久、undo 回滚、binlog 复制
优化：慢日志→explain→索引/改写
```

## 项目名称：对内 LLM 网关 + 异步任务服务

| 字段 | 填写示例 |
|------|----------|
| 业务一句话 | 给业务系统统一接入大模型：鉴权、限流、超时、流式输出；长任务异步化，避免拖垮 Web 进程 |
| 你的角色 | Python 后端，负责 FastAPI 网关、异步调用 LLM、队列消费与基础观测 |
| 规模数据 | 日均请求约 **1.2 万**；峰值约 **80 QPS**；单次生成 P95 目标 &lt; 15s（流式首包 &lt; 1.5s） |
| 技术栈 | Python 3.11、FastAPI、asyncio、httpx、Redis、（可选）Celery/RQ、Prometheus 指标 |
| 最大难点 1 | 在协程里误用阻塞 IO / 同步 SDK，把事件循环卡死，延迟毛刺与吞吐双崩 |
| 方案与取舍 | 全链路 async；阻塞库丢 `asyncio.to_thread`；写操作用队列+幂等；**不**在 request 内硬等长链路批处理；多租户限流在网关层 |
| 结果 | 事件循环卡死工单 **0**；P95 从偶发 40s+ 收到约 **12s**；模型费用看板可按业务线拆分 |
| 复盘 | 初期缺统一 timeout/重试策略；应用默认参数坑导致多 worker 共享问题，已改工厂函数 |

### STAR 口述稿（约 3 分钟）

- **S：** 多条业务线各自调模型，超时、重试、账单混乱；有同步 SDK 堵死 worker。  
- **T：** 我负责统一 Python 网关：鉴权限流、统一超时、流式、长任务异步，并保证协程模型正确。  
- **A：** ① FastAPI async 路由 + httpx.AsyncClient 调模型。② 发现同步 SDK 后统一 `to_thread` + 信号量限并发。③ 长摘要类任务改「提交任务→轮询/回调」，Redis 存状态，幂等 task_id。④ 打点：QPS、首包延迟、错误码、token 费用。  
- **R：** 毛刺下降、费用可归因；形成内部接入规范（超时/重试/幂等）。  

### 面试官可能追问（示例答）

1. **GIL 下为什么还能抗 80 QPS？**  
   主要是 IO 等待；瓶颈在模型 RT 与连接池，不在 CPU 算字节码。CPU 重活另进程或 C 扩展。  
2. **协程里能不能用 time.sleep？**  
   不能；用 `asyncio.sleep`。同步阻塞会卡住整个 loop 上的所有请求。  
3. **失败重试会不会重复扣费/重复写？**  
   读可重试；写带幂等键；模型侧尽量用业务 request_id；状态机只允许合法迁移。

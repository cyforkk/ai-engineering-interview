# 一、超高频

## 1. 基础概念与场景

### 1.1 为什么用消息队列？优缺点？

**优点（三大核心）：**

| 作用 | 含义 |
|------|------|
| **解耦** | 生产消费互不知对方细节 |
| **异步** | 非核心链路后置，降接口 RT |
| **削峰** | 峰值进队列，按能力消费 |

**缺点 / 带来问题：**

- 系统**可用性**多一依赖  
- **复杂性**上升（运维、排查）  
- **一致性**变最终一致  
- 重复消费、顺序、积压要治理  

---

## 2. 消息可靠性（最核心）

### 2.1 如何保证消息不丢失？

| 环节 | 手段 |
|------|------|
| **生产者** | 发送确认（Kafka acks、Rabbit confirm）；失败重试；本地消息表 |
| **Broker** | 持久化；多副本（Kafka 多 replica）；刷盘策略 |
| **消费者** | 业务处理成功后再 ACK/提交 offset；禁用自动提交慎用 |

```text
生产确认成功 → Broker 落盘/多副本 → 消费成功再提交进度
```

### 2.2 如何保证不重复消费？（幂等）

- 网络重试导致 **至少一次** 最常见 → 消费必须幂等  
- 手段：  
  - 业务**唯一 ID**（消息 key / 业务单号）  
  - DB **唯一约束**  
  - 状态机（已处理跳过）  
  - Redis setnx 去重（注意 TTL）  

### 2.3 如何保证顺序？

- **单分区 / 单队列** + 同 key 进同分区 + 单线程消费该分区  
- 全局顺序代价大；业务局部有序即可  
- 或业务层带序号排序（慎用）  

### 2.4 消息积压怎么办？

1. **扩容消费者**（先保证分区/队列够并行）  
2. 临时加机器 / 提高并发  
3. **优化消费**（批量、异步、降 DB 压力）  
4. 降级：跳过非关键、另写冷队列  
5. 紧急：扩分区（Kafka 只能增、注意 hash）

## 3. Kafka / RocketMQ / RabbitMQ 对比

### 3.1 对比表（必背维度）

| 维度 | Kafka | RocketMQ | RabbitMQ |
|------|-------|----------|----------|
| 吞吐 | **极高** | 高 | 中 |
| 延迟 | 可低 | 低 | 较低 |
| 可靠性 | 副本、acks | 主从/DLedger 等 | 镜像队列等 |
| 功能 | 流/日志强 | 事务/延时/业务特性多 | 路由灵活 |
| 运维 | 偏大数据/日志 | 国内业务友好 | 传统企业多 |
| 场景 | 日志、大数据、高吞吐 | 电商订单、事务消息 | 复杂路由、传统 MQ |

### 3.2 Kafka 为何吞吐高？

1. **顺序写磁盘**（追加日志）  
2. **零拷贝**（sendfile 等）  
3. **分区并行**  
4. **批量发送/拉取**  
5. 页缓存  

### 3.3 RocketMQ 事务消息

```text
1. 发「半消息」到 Broker（消费者不可见）
2. 执行本地事务
3. 成功 → Commit；失败 → Rollback
4. 未知 → Broker 回查生产者本地事务状态
```

- 用于最终一致：订单 + 发消息对齐。  

### 3.4 RabbitMQ 交换机

| 类型 | 路由 |
|------|------|
| Direct | routing key 精确匹配 |
| Fanout | 广播所有绑定队列 |
| Topic | 通配符匹配 |
| Headers | 头属性匹配（少用） |

## 4. 其他超高频点

### 4.1 死信队列 DLQ

- 消费多次失败 / 过期 / 拒绝 的消息进入死信  
- 用途：隔离坏消息、人工补偿、告警  

### 4.2 延迟消息

| 方案 | 说明 |
|------|------|
| RocketMQ | 延时/定时消息（级别或定时） |
| RabbitMQ | TTL + 死信 / 延迟插件 |
| Kafka | 无原生，用时间轮/业务或外部调度 |

### 4.3 分布式事务与 MQ

- RocketMQ **事务消息**  
- 本地消息表 + 定时投递  
- 最大努力通知、Saga 等最终一致

# 二、高频（MQ 进阶）

## 5. Kafka 要点

| 概念 | 含义 |
|------|------|
| **ISR** | 与 leader 同步的副本集合 |
| **LEO** | Log End Offset，日志末端 |
| **HW** | High Watermark，消费者可见上限（ISR 最小 LEO 等规则） |
| 分区 | 并行单位；有序以分区为界 |
| 副本 | 高可用；acks=all 更安全 |
| Consumer Group | 组内分摊分区，组间广播式独立 |

### 分配与均衡

- 分区数 ≥ 消费者数才吃满并行  
- rebalance：消费者变更时重新分配（要稳定消费）  

### 回溯

- 按 offset / 时间重置消费位点（reset）  

## 6. RocketMQ 架构

- **NameServer**：轻量路由注册（无强依赖 ZK）  
- **Broker**：存消息  
- **Producer / Consumer**  
- 消费：集群模式分摊；广播模式全收  

---

# 三、中频 / 低频（MQ）

- Exactly-Once：幂等生产 + 事务/幂等消费 + 精确提交（Kafka 事务等），成本高  
- 监控：积压 lag、消费延迟、失败率、重试次数  
- KRaft：Kafka 去 ZK  
- RocketMQ 5：Pop 消费、分层存储等（了解）

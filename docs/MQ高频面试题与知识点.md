## 二、P0 必会

### 2.1 为什么用 MQ

| 收益 | 代价 |
|------|------|
| 解耦、异步降 RT、削峰 | 复杂、延迟、一致难、运维排查成本 |

**高频题：** 为什么引入？不用行不行？带来什么问题？

---

### 2.2 消息不丢（三段）

| 阶段 | 手段 |
|------|------|
| 生产 | 发送确认、失败重试 |
| Broker | 持久化、副本 |
| 消费 | **业务成功后再 ack** |

常见语义：**至少一次** → 必须幂等。

**高频题：** 如何保证不丢？消费者挂了？

---

### 2.3 重复与幂等

**原因：** 超时重试、重复投递、ack 前崩溃。  

**手段：** 业务唯一键 + DB 唯一索引；状态机；Redis setnx；去重表。

**高频题：** 为何重复？如何幂等？下单扣款怎么防重？

**口述：**  
> 用至少一次保证不丢，用业务幂等消化重复。

---

## 三、P1 常考

### 3.1 顺序

- 全局有序：单队列，吞吐差  
- 局部有序：同 key 同分区 + 单线程消费  

### 3.2 积压

1. 扩消费者 / 提并行  
2. 优化消费、批量  
3. 查上游突增；Kafka 可扩分区  
4. 监控堆积滞后  

### 3.3 最终一致

| 方案 | 点 |
|------|-----|
| 本地消息表 | 与业务同事务 |
| 半消息/事务消息 | RocketMQ 经典 |
| 最大努力通知 | 回调重试 |

## 四、P2 加分

| 点 | 一句话 |
|----|--------|
| 死信 | 多次失败进 DLQ |
| 延迟消息 | 超时关单 |
| Kafka | 分区、副本、消费组、offset |
| 选型 | 吞吐→Kafka；业务事务/延时→RocketMQ；路由→Rabbit（按简历深挖一种） |

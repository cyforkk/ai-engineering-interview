## 一、本专题在考什么

1. 性能从哪来（才能判断「该不该上 Redis」）  
2. 缓存三连能否**定义分开**并讲方案动机  
3. 分布式锁每一步参数的**必要性**  
4. 与 DB 一致性为何选主流做法  

---

## 二、为什么 Redis 快

| 因素 | 所以然 |
|------|--------|
| 内存 | 随机访问延迟远低于磁盘；热数据放内存是数量级差异 |
| 命令单线程执行 | 避免共享可变状态上的锁竞争与切换；逻辑简单正确 |
| 高效结构 | SDS、跳表、压缩列表等针对场景优化 CPU 与内存 |
| IO 多路复用 | 一线程盯多连接，不为一连接阻塞整站 |

**注意：** Redis 6+ 多线程主要加速**网络读写**，不要说成「多线程并行执行任意命令」。命令执行路径仍以单线程模型理解更安全。

### 五类型：为何不是只有 String

不同访问模式需要不同复杂度保证：排行需要有序+按分查 → ZSet（跳表等）；去重集合运算 → Set；字段局部更新 → Hash，避免每次整 JSON 读写。

## 三、缓存三连（必须讲清差异）

### 穿透：为什么会发生

缓存只对「曾经写过的 key」有意义。攻击或乱查**永远不存在**的 id，缓存永远 miss，流量打穿到 DB → DB 被拖垮。

**为何空值缓存有用：** 让「不存在」也成为可命中的结果，把打穿变成命中；必须短 TTL，防止恶意 key 占满内存。布隆过滤器在更前拦截「一定不存在」。

### 击穿：为什么是「一个热点」

热点 key 过期瞬间，成千上万请求同时回源重建，像针尖扎穿。  
与穿透差别：**数据本来存在**，只是失效窗口太大。

**互斥重建：** 只让一个请求加载 DB，其它等待或稍后重试——用正确性换 DB 保护。  
**逻辑过期：** 不真正删除，value 带逻辑过期时间，过期仍返回旧值并异步刷新——用短暂脏读换稳定。

### 雪崩：为什么是「一片」

大量 key **同一时刻**过期，或 Redis 集群故障，回源风暴或服务降级。  
TTL 加随机抖动：把过期曲线打散。高可用与多级缓存：降低「Redis 没了就全死」的概率。限流：保护 DB 活着。

## 四、分布式锁：每个参数为什么存在

```text
SET key unique_value NX EX seconds
```

| 要素 | 没有会怎样 |
|------|------------|
| NX | 无法互斥，多实例同时进临界区 |
| EX | 持锁进程崩溃 → 死锁 |
| unique value | 超时后 A 的锁过期，B 加锁，A 回来 del 会删掉 **B 的锁** |
| Lua 删 | get 与 del 非原子时仍有误删窗口 |

**续期：** 业务可能超过 EX，不续期会误释放。Redisson 看门狗是工程化续期。  
**主从风险：** 异步复制下主挂可能丢锁；要能讲清业务能否接受，而不是背「RedLock 绝对正确」。

---

## 五、Cache Aside：为何先 DB 再删缓存

目标：读多写少时，缓存是 DB 的加速视图。

- **写先 DB：** 以 DB 为权威真相  
- **再删缓存：** 下次读 miss 回源拿到新数据  

若先改缓存再写 DB：DB 失败时缓存已是新值 → 长期脏。  
若更新缓存而不是删：并发读写下更难推理「最终以谁为准」。删缓存是更简单的失效策略；删失败用重试/ binlog 补偿。强一致另论，成本高。

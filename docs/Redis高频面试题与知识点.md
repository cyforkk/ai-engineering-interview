## 1. Redis 是啥？为啥快？

### 先记一句

Redis 是**内存里的超快小仓库**，常放「热点数据」，减轻数据库压力。

### 生活例子

- 数据库 = 大仓库（全、稳、慢一点）  
- Redis = 收银台旁边的**常卖货小货架**（近、快，但只放一部分）

### 为啥快？（四点就够）

1. **在内存里干活**——比读写磁盘快很多  
2. **执行命令基本单线程**——少抢锁、逻辑简单  
3. 数据结构自己做得很高效  
4. 一个线程可以盯很多网络连接（多路复用）

注意：别说「Redis 命令随便多线程并行跑」。网络收发可以多线程优化，**命令执行**你仍按单线程理解最稳。

### 五种类型记场景

| 类型 | 你可以把它想成 | 典型用途 |
|------|----------------|----------|
| String | 一个标签贴一句话 | 缓存、计数、锁 |
| Hash | 一张小表格 | 用户资料多个字段 |
| List | 一队人 | 最新消息列表 |
| Set | 不重复的一堆 | 点赞人、去重 |
| ZSet | 带分数的排名 | **排行榜**、按时间排序 |

## 2. 缓存三连：最容易混，用故事分开

### 穿透：查一个「根本没有的人」

有人拿假身份证号狂查。  
缓存没有（从来就没有）→ 每次都打到数据库 → 数据库被拖垮。

**怎么办？**

- 把「查无」也缓存一小会儿（空值缓存）  
- 或门口放个「大概有没有」的快速检查（布隆过滤器）  
- 参数先校验（id 明显非法直接拒）

### 击穿：超级网红商品，标签刚好过期

这个商品**其实有**，但缓存刚好过期那一秒，所有人一起冲进仓库进货 → 仓库炸。

**怎么办？**

- 同一时间只允许一个人去数据库重建缓存，别人等一会儿  
- 或「逻辑过期」：过期了先把旧的给你，后台偷偷更新

### 雪崩：很多标签同一时间掉光，或小货架塌了

一大片缓存一起失效，或 Redis 挂了，所有流量砸向数据库。

**怎么办？**

- 过期时间加一点随机，别整齐划一到期  
- Redis 高可用、限流、本地再挡一层  

### 三个差在哪？（考试就考这个）

| | 查的东西 | 像什么 |
|--|----------|--------|
| 穿透 | **从来不存在** | 假身份证狂刷 |
| 击穿 | **热点突然过期** | 网红商品标签掉了 |
| 雪崩 | **一大片一起挂** | 货架集体空了/倒了 |

## 3. 分布式锁：为啥要那么多参数？

### 先记一句

多台机器时，Java 的 `synchronized` 只锁**自己那台**。要用 Redis 当「公共厕所的门锁」。

### 正确姿势（记住命令形态）

```text
SET lock_key 随机唯一值 NX EX 30秒
```

| 参数 | 白话 | 没有会怎样 |
|------|------|------------|
| NX | 没有锁才加成功 | 大家一起进门 |
| EX | 最多锁多久 | 人晕倒了门永远锁死 |
| 随机唯一值 | 证明锁是我的 | 我超时后误删别人的锁 |
| 用 Lua 删锁 | 先确认是我的再删，一步完成 | 中间被插队，误删 |

业务特别长：要**续命**（锁快过期再续一段时间），否则锁没了别人进来。

## 4. 和数据库谁先谁后？

### 常见写法（Cache Aside）

1. **读：** 先看 Redis，没有再查库，再放回 Redis  
2. **写：** **先改数据库，再删 Redis**（下次读会重新加载）

### 为啥不先改缓存？

数据库才是「账本真相」。先改缓存、库没改成功，就会长期是假数据。  
删缓存更简单：当缓存暂时失效，下次读以库为准。

删失败？重试或消息补偿。业务大多接受「很快一致」，不是银行账户那种绝对同时。

## 1. Redis 为什么快？

- 数据在**内存**。
- **命令执行**单线程，无锁竞争（逻辑简单）。
- 高效数据结构。
- **IO 多路复用**处理大量连接。
- **注意**：Redis 6+ 多线程主要优化网络读写，不是「命令随意多线程并行」。

---

## 2. 五种基本类型及场景？

- **String**：缓存、计数、Session、分布式锁。
- **Hash**：对象多字段（局部更新）。
- **List**：最新列表、简单队列。
- **Set**：去重、共同关注。
- **ZSet**：排行榜、按 score 排序/延时。

---

## 3. 缓存穿透 / 击穿 / 雪崩？（必考，分开说）

- **穿透**
  - 定义：查**根本不存在**的数据，缓存不中，每次打 DB。
  - 方案：缓存空值（短 TTL）、布隆过滤器、参数校验。
- **击穿**
  - 定义：**热点 key** 过期瞬间，大量并发回源。
  - 方案：互斥锁重建、逻辑过期、永不过期 + 异步刷新。
- **雪崩**
  - 定义：大量 key **同时过期**，或 Redis 宕机。
  - 方案：TTL 加随机、集群高可用、限流降级、多级缓存。

---

## 4. 分布式锁怎么做？（必考）

- 命令：`SET key unique_value NX EX seconds`
  - **NX**：不存在才成功（互斥）。
  - **EX**：过期防死锁。
  - **unique_value**：标识持有者，防误删别人的锁。
- 删除：Lua 脚本「校验 value 再 del」，保证原子。
- 长任务：需要**续期**（如 Redisson 看门狗）。
- 风险：主从切换极端情况可能丢锁；业务需评估。

---

## 5. 缓存与 DB 一致性？

- 常见 **Cache Aside**
  - 读：先缓存，miss 再 DB，回填缓存。
  - 写：**先更新 DB，再删除缓存**。
- 删缓存失败：重试 / binlog 补偿。
- 多最终一致；强一致成本高。

---

## 6. 持久化 RDB / AOF？

- **RDB**：快照，恢复快，可能丢最近一段时间数据。
- **AOF**：记写命令，更耐丢数据，文件可能更大；可重写压缩。
- 可混合使用。

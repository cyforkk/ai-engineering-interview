## 二、为什么快 + 类型（P0）

### 白话

内存计算、命令执行单线程（少锁竞争）、高效结构、IO 多路复用。  
（Redis6 多线程主要优化网络读写，别说成「命令多线程随便并行」。）

| 类型 | 场景 |
|------|------|
| String | 缓存、计数、Session、锁 |
| Hash | 对象字段 |
| List | 最新列表 |
| Set | 去重、共同好友 |
| ZSet | **排行榜**、延时 score |

### 面试怎么答

> 四点为什么快 + 五类型各举一场景。

## 三、缓存三连（P0 最易混）

| 问题 | 白话 | 方案方向 |
|------|------|----------|
| **穿透** | 查根本不存在的，每次打穿到 DB | 空值缓存、布隆、参数校验 |
| **击穿** | **热点** key 刚好过期，并发打爆 DB | 互斥重建、逻辑过期、永不过期+异步刷 |
| **雪崩** | 大量 key 同时过期或 Redis 挂了 | TTL 打散、集群高可用、限流、多级缓存 |

### 面试怎么答

> 先定义三者不同，再各给方案；不要背成一套话。

## 四、分布式锁（P0）

### 白话

多实例下 JVM 锁锁不住别人机器。用 Redis：

```text
SET key unique_value NX EX seconds
```

- NX：没有才成功（互斥）  
- EX：超时防死锁  
- unique：删除时确认是自己的锁  
- 删除用 **Lua** 校验+del 原子  

长任务要续期（Redisson 看门狗）。主从切换可能丢锁，要说清业务能否接受。

**完整稿：** [Redis面渣 · 锁](./Redis面渣级口述.md)

## 五、与 DB 一致性（P1）

常用 **Cache Aside：先更新 DB，再删缓存**（不是先改缓存）。  
删失败要重试或 binlog 补偿。强一致很难，业务多最终一致。

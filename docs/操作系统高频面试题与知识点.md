## 一、本专题要搞定什么

## 二、进程 vs 线程（P0）

### 白话

- **进程**：操作系统分配资源的基本单位。每个进程有独立地址空间（自己的「内存地盘」），互相默认隔离，A 崩了一般不直接写坏 B 的内存。
- **线程**：调度的基本单位（CPU 时间片切给谁）。同一进程里的线程 **共享堆、全局变量**，通信便宜，但共享数据就要同步（锁），否则竞态。
- **Java 视角**：`new Thread` / 线程池里的线程，共享同一 JVM 堆 → 所以 HashMap 多线程会出事，要 ConcurrentHashMap 或同步。

### 面试怎么答（可直接照着说，约 1 分钟）

> 进程是资源分配单位，有独立地址空间，隔离性强；线程是 CPU 调度单位，同进程线程共享堆和全局数据，切换开销通常比进程小。  
> 优点是通信快、创建轻；代价是一个线程写坏共享数据会影响全进程，必须做同步。  
> 在 Java 里所有业务线程共享堆，所以集合、单例、缓存都要考虑线程安全；Redis 单线程执行命令也是用「少共享可变状态」换简单正确。

### 追问

| 问 | 答要点 |
|----|--------|
| 进程通信方式？ | 管道、消息队列、共享内存、Socket 等 |
| 协程和线程？ | 用户态调度，更轻；Python asyncio / Go goroutine 思路相近 |
| 为何线程池？ | 复用，避免反复创建销毁；控制并发度 |

## 三、用户态 / 内核态 / 系统调用（P0）

### 白话

普通程序跑在 **用户态**，不能随便碰硬件、改页表。要读文件、建网络连接，必须走 **系统调用** 进入 **内核态**，由内核代劳。  
每次切换有成本（保存寄存器、权限检查）。所以高性能 IO 会：批量读写、缓冲、mmap、零拷贝，减少「用户↔内核」来回次数。

### 面试怎么答

> 用户程序不能直接访问硬件，需要通过系统调用陷入内核。  
> 用户态到内核态的切换有开销，所以高 QPS 服务会减少无意义的 syscall，用缓冲、批量、epoll 多路复用、必要时零拷贝。  
> 比如频繁小 write 不如缓冲后批量；Redis/Nginx 用事件驱动少阻塞在 syscall 上。

### 追问

| 问 | 答要点 |
|----|--------|
| 常见 syscall？ | read/write/open/socket/epoll_wait… |
| 零拷贝？ | 减少内核与用户空间数据拷贝次数（sendfile 等） |

## 四、上下文切换（P0）

### 白话

CPU 从一个线程切到另一个：要保存当前寄存器、程序计数器，再恢复目标线程状态。  
切换本身不是免费的；**切换太频繁**会出现：CPU 占用高、业务吞吐却上不去（都在「换人干活」）。

### 面试怎么答

> 上下文切换是调度时保存/恢复线程运行现场。  
> 过多切换会浪费 CPU。工程上用线程池限制线程数、用异步/事件驱动减少阻塞线程数量，避免「线程爆炸」。

## 五、epoll 与 IO 多路复用（P1，后端高频）

### 白话

一台机器要服务上万连接。若 **一个线程阻塞读一个连接**，线程会爆。  
**IO 多路复用**：一个线程监听很多 fd，谁就绪（有数据可读/可写）再处理谁。

- **select/poll**：每次把关注列表交给内核，内核线性扫，连接多了慢。  
- **epoll**（Linux）：用就绪队列，活跃连接多时更合适。  
Nginx、Redis、Netty 都是这类思路。

### 面试怎么答（约 1 分钟）

> 高并发网络服务不能一连接一线程阻塞读。  
> 用 IO 多路复用：一个线程通过 epoll 监听大量连接，只处理就绪的那些。  
> select/poll 每次要扫大量 fd；epoll 维护就绪列表，适合 C10K 以上场景。  
> 我理解 Redis 单线程处理命令、Nginx worker 事件循环，底层都受益于这种模型。

### 追问

| 问 | 答要点 |
|----|--------|
| 水平触发 vs 边缘触发？ | LT 只要还有数据就通知；ET 状态变化才通知，要一次读干净 |
| 和 Java NIO 关系？ | Selector 多路复用思想一致 |

## 六、死锁（P0）

### 白话

多个线程互相拿着对方需要的锁，都在等，谁也走不了。

**四个必要条件（缺一不可同时成立才可能死锁）：**

1. 互斥  
2. 占有且等待  
3. 不可抢占  
4. 循环等待  

**破：** 统一加锁顺序、超时获取、减少锁粒度、死锁检测后回滚一方（DB 常见）。

### 面试怎么答

> 死锁是并发方循环等待资源。四条件：互斥、持有等待、不可抢占、环路。  
> 预防常用固定加锁顺序和超时；线上可用 jstack 看 Java 线程是否互相 BLOCKED；数据库 InnoDB 会检测死锁并回滚一方。

## 七、虚拟内存与缺页（P1）

### 白话

进程以为自己有连续大内存，其实是 **虚拟地址**；内核用页表映射到物理页。  
访问到没加载的页 → **缺页异常** → 内核从磁盘/交换区装入，慢。  
JVM 对象在进程虚拟地址空间里；容器还受 cgroup 内存限制，RSS 超了可能被 OOM Kill。

### 面试怎么答

> 虚拟内存通过页表把虚拟地址映射到物理内存，实现隔离和超卖。  
> 缺页会走磁盘，延迟高。  
> 调 JVM 堆时要同时看宿主机/容器 limit，堆外和 page cache 也占物理内存。

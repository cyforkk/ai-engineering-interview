## 一、并发到底在解决什么问题

单核也能「并发」（时间片轮转），多核可以「并行」。难点不在「开线程」，而在：

1. **原子性**：多步操作中间被打断会不会看到半成品  
2. **可见性**：A 线程写的值 B 能否及时看到  
3. **有序性**：指令重排后语义是否仍符合程序员直觉  

CPU 缓存、编译器优化都是为了**快**；JMM（Java 内存模型）用规则告诉你：在什么同步手段下，跨线程语义可预期。

## 二、happens-before：可见性的「法律条文」

若没有规范，编译器/CPU 可以在单线程语义不变的前提下重排指令；但对多线程，重排可能让 B 永远看不到 A 的写。

**happens-before（hb）**：若操作 A hb 操作 B，则 A 的写对 B 可见，且 A 的顺序在 B 之前（从内存语义角度）。

常见建立 hb 的方式：

| 规则 | 含义 |
|------|------|
| 程序次序 | 单线程内前面的 hb 后面的 |
| 监视器锁 | unlock hb 后续对同一锁的 lock |
| volatile | 写 hb 后续对同一变量的读 |
| 线程 start/join | start 前写对线程内可见；join 后线程内写对 join 后可见 |
| 传递性 | A hb B 且 B hb C ⇒ A hb C |

**所以然：** 你用的 `synchronized`/`volatile`/`concurrent` 工具，底层都在建立 hb，而不是「魔法让变量同步」。

## 三、volatile：能做什么、不能做什么

### 机制直觉

写 volatile：插入内存屏障，把工作内存刷新到主内存相关可见路径。  
读 volatile：使后续读不会用过期缓存值（简化理解）。

### 为什么不保证 i++

`i++` 是读-改-写三步。两线程可交错成丢更新。volatile 只保证单次读写的可见与有序，**不加锁就不给复合操作原子性**。

### 适用

- 状态标志：`shutdown = true`  
- 安全发布（配合正确双检锁模式中的实例引用）  
- 一个线程写、多线程读的简单发布

## 四、synchronized 与 Lock

### synchronized 为什么能保证可见+原子

进入/退出监视器时建立 hb：解锁前的写对后续加锁的线程可见；互斥保证临界区不交叉执行。

锁升级（偏向→轻量→重量）是 JVM 实现优化，面试能提「自适应降低无竞争开销」，细节不必背字节码级别。

### ReentrantLock / AQS

显式锁提供：可中断、tryLock 超时、公平可选、多个 Condition。  
**AQS：** 用 `state` 表示同步状态 + CLH 风格等待队列。子类实现 tryAcquire/tryRelease，队列阻塞唤醒复用。  
**所以然：** 锁、信号量、CountDownLatch 结构相似，抽公共骨架避免重复。

可重入：同一线程再次加锁 `state++`，防止自己堵自己。

## 五、线程池：每个参数的存在理由

| 参数 | 所以然 |
|------|--------|
| corePoolSize | 常驻线程，承接稳态负载，避免反复创建 |
| maximumPoolSize | 突发时上限，防止无限创建打爆内存 |
| workQueue | 缓冲任务；**有界**才能把压力传导到拒绝策略 |
| keepAliveTime | 非核心线程空闲回收，省资源 |
| handler | 系统过载时明确失败/降级，而不是静默丢光 |

### 提交顺序为什么是 core → 队列 → max

先希望用核心线程稳稳干活；忙不过来先排队（吸收毛刺）；队列也满再扩到 max；再满则拒绝。  
若队列**无界**，任务永远进队，max 线程可能永远不创建，最后 OOM——这是经典误配。

### 拒绝策略

- Abort：抛异常，调用方感知  
- CallerRuns：调用线程自己跑，反向施压  
- Discard / DiscardOldest：丢弃，要业务能接受

## 六、ThreadLocal

每个线程一个副本，避免传参与共享。  
**泄漏原因：** 线程池线程不销毁，ThreadLocalMap 条目长期活着；value 占大对象。必须 `try/finally remove`。

---

## 七、CAS 与原子类

CAS：CPU 级比较交换，无锁更新。失败则重试（自旋）。  
ABA：值从 A→B→A，CAS 以为没变；用版本号或 `AtomicStampedReference`。  
`AtomicInteger` 等封装 CAS，适合热点计数；竞争极高时再考虑 LongAdder 分段。

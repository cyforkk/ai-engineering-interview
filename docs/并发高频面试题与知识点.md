## 这个专题在问啥？

多线程一起干活时：

1. 会不会看错数？（可见性）  
2. 会不会步骤做一半被打断？（原子性）  
3. 线程池怎么配才不会把自己打爆？

---

## 1. 先建立感觉：三个坑

### 生活例子：合写文档

两个人同时改同一份在线文档：

- A 改了标题，B 屏幕还显示旧的 → **看不见最新**（可见性）  
- 两人都点「点赞+1」，结果只加了一次 → **步骤交叉**（原子性）  
- 系统自动重排保存顺序，搞出怪结果 → **顺序被打乱**（有序性）

Java 用锁、`volatile`、并发工具来管这些坑。

---

## 2. synchronized：最常用的「进门上锁」

### 先记一句

`synchronized` = 同一时刻**只让一个线程进这段代码**（同一把锁的前提下）。

### 生活例子

公共卫生间：有人进去反锁，别人等；出来下一个人进。  
出来的时候，相当于把你在里面做的改动「公开」给后面的人（可见性也有了）。

---

## 3. volatile：只能做信号灯，不能当柜台

### 先记一句

`volatile` 适合：**一个开关**，大家都能尽快看到最新开/关。  
**不适合** `i++` 这种「先读再改再写」。

### 例子

- `running = false` 通知线程停下来 → 用 volatile 很常见  
- 两个线程同时 `count++` → 即使用 volatile，仍可能丢次数，要加锁或用原子类  

**为啥 i++ 不行？**  
因为是三步：取出 → 加一 → 写回。两人交错就会算错。volatile 不管这三步捆在一起。

---

## 4. 线程池：为啥不要 new Thread 飞？

### 生活例子

饭店不能每来一个客人就新招一个厨师（雇人贵、厨房挤爆）。  
正确做法：**固定几个厨师 + 出餐口排队 + 人太多就婉拒或让顾客自己端一点**。

| 参数 | 人话 |
|------|------|
| 核心线程数 | 常驻厨师 |
| 最大线程数 | 忙时最多雇到几个 |
| 队列 | 出餐口最多堆多少单 |
| 拒绝策略 | 堆满了怎么办：报错 / 让提交的人自己干 / 丢掉 |

### 任务怎么走？（记住顺序）

1. 厨师还不满 → 先上人  
2. 人满了 → 先排队  
3. 队也满了 → 再临时加人到最大  
4. 还满 → 拒绝策略  

**坑：** 队列如果无限长，任务会堆到内存爆，临时工（最大线程）可能永远用不上。

---

## 5. ThreadLocal 一句话

每人一本**自己的小本子**，互不干扰。  
但线程池会**重复用同一批人**，用完不擦小本子，下一个客人可能看到上一个人的字 → 要 `remove`。

---

## 6. CAS / 原子类（了解）

「我看还是 5，才改成 6」——比较并交换。失败就再试。  
适合简单计数；竞争特别惨烈时有更合适的工具（如 LongAdder）。

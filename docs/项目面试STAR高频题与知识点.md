## 二、P0：STAR 框架

| 字母 | 含义 | 要说清 |
|------|------|--------|
| **S** Situation | 背景 | 业务、用户、你的角色 |
| **T** Task | 任务 | 目标指标（RT、稳定性、成本） |
| **A** Action | 行动 | **你**做了什么（技术点+步骤） |
| **R** Result | 结果 | 数字前后对比 + 沉淀 |

**原则：**

- 用「我」少用「我们」含糊带过  
- 每个项目准备 **1 个主故事 + 1 个事故/优化**  
- 技术词能落到代码/架构图一句话

## 三、P0：一分钟项目介绍结构

```text
1. 项目是干什么的（10s）
2. 技术栈与体量：用户量/QPS/数据量（15s）
3. 你负责哪个模块（15s）
4. 最有含金量的一点（难点或结果）（20s）
```

## 四、P0 高频深挖题清单

### 4.1 通用

1. 介绍一个你最有成就感的项目  
2. 项目中最难的技术点？怎么解决的？  
3. 为什么选 A 不选 B？  
4. 有没有量化结果？  
5. 如果重做会怎么改？  
6. 和同事意见不一致怎么办？  
7. 你在团队里的角色？带人吗？  
8. 遇到过线上故障吗？  

### 4.2 后端技术绑定（按简历勾选准备）

| 简历点 | 可能追问 |
|--------|----------|
| Redis | 穿透击穿？一致性？大 key？ |
| MySQL | 慢 SQL？索引？事务？分库？ |
| MQ | 丢消息？重复？积压？顺序？ |
| 线程池 | 参数？拒绝？隔离？ |
| 接口慢 | 怎么排查分层？ |
| 分布式锁 | 实现？续期？误删？ |
| Spring 事务 | 失效过吗？ |
| 微服务 | 雪崩？限流？链路？ |
| JVM | Full GC？OOM？ |

## 五、指标怎么说（P0）

| 类型 | 示例 |
|------|------|
| 性能 | P99 从 800ms → 120ms |
| 容量 | 支撑 QPS 从 500 → 3000 |
| 稳定性 | 故障次数、可用性 99.9% |
| 成本 | 机器数下降、RT 降带来的体验 |
| 业务 | 转化率、完单率、客服解决率 |

**没有精确数：** 用合理区间 + 对比（「高峰约两千 QPS」「超时率从明显有到基本没有」），别编造离谱数据。

## 七、P1：常见「坑」答法

| 坑 | 更好的说法 |
|----|------------|
| 只罗列技术栈 | 用 STAR 讲一个闭环 |
| 全是「我们」 | 拆清你独立负责部分 |
| 没有结果 | 至少给定性+过程指标 |
| 贬低前司/同事 | 客观说约束与取舍 |
| 简历写了不会 | 删掉或当天补到能讲 |

## 一、本专题在考什么

面试官表面上在问 `equals`、`String`、`HashMap`，实际在考三层能力：

1. **语言契约**：对象相等、哈希、不可变——这是集合正确性的前提  
2. **数据结构取舍**：数组、链表、树各自解决什么问题、付出什么代价  
3. **工程直觉**：线程安全、扩容、迭代失败——能不能接到线上事故  

---

## 二、==、equals、hashCode

### 是什么

| 概念 | 含义 |
|------|------|
| `==` | 基本类型比**值**；引用类型比**是不是同一个对象**（引用相等） |
| `equals` | `Object` 默认实现也是引用相等；子类可重写为**业务相等**（内容） |
| `hashCode` | 把对象映射成 `int`，供哈希表**先分桶** |

### 为什么需要 equals 契约

哈希表查找分两步：**先用 hash 找桶，再在桶内用 equals 确认**。  
若两个对象业务上相等（equals true），但 hashCode 不同，它们会进**不同桶**——`put` 进去的对象 `get` 时可能永远找不到。这不是风格问题，是**正确性**问题。

因此契约是：

- equals 为 true ⇒ hashCode **必须**相同  
- hashCode 相同 **不** ⇒ equals 一定 true（允许碰撞）

### 为什么默认 equals 用引用

通用对象没有「业务主键」概念；语言只能保守地按身份比。业务类（用户、订单）自己定义相等维度，并**成对**重写 equals/hashCode（或用 `record`/`Objects` 工具）。

### 边界

- 只用 IDE 生成 equals 却漏字段 → 逻辑错  
- 可变对象当 HashMap key 且改了参与 hash 的字段 → 丢数据  
- `hashCode` 分布差 → 碰撞多 → 链表/树变长 → 变慢

## 三、String 为什么设计成不可变

### 是什么

`String` 内容创建后不可改：类 `final`，内部字符数据不提供对外修改 API。`s = s + "x"` 是**新对象赋值**，不是原地改。

### 为什么这样设计（所以然）

1. **常量池可共享**  
   字面量可放入池，多处引用同一实例，省内存。若可变，一处修改会污染所有共享方——池化立刻不安全。

2. **hashCode 可缓存**  
   String 常作 HashMap key。不可变 ⇒ 内容不变 ⇒ hash 算一次可缓存。若可变，每次改内容都要让哈希表重新定位，几乎不可用。

3. **安全与线程安全**  
   类名、文件路径、网络参数、鉴权 token 若可被中途改写，安全模型崩溃。不可变天然无数据竞争。

### 拼接为什么用 StringBuilder

`+` 在循环里可能反复创建中间对象（编译器有时优化，不可依赖）。`StringBuilder` 在可变缓冲上追加，最后再 `toString`。`StringBuffer` 方法带同步，现代单线程拼接触用 Builder 即可。

### 边界

- `new String("a")`：字面量可能已在池；`new` 再在堆上造一个  
- `intern`：尝试返回池中引用（了解即可，别滥用）

## 四、HashMap（核心）

### 是什么

键值哈希表：平均 O(1) 级别查找/插入（在 hash 良好时）。JDK 8：`Node` 数组 + 链表，冲突严重时树化为红黑树。

### 为什么用哈希而不是每次遍历 List

数组按下标 O(1)，但键是任意对象，不能直接当地址。  
hash 把键**映射**到下标；冲突不可避免，所以桶内还要链表/树处理「多个键映到同一下标」。

### 为什么容量是 2 的幂

下标计算用 `(n - 1) & hash` 代替 `% n`：位运算更快，且在 2 的幂长度下等价于取模，分布相对均匀。扩容时也有「元素要么在原下标、要么在原下标+旧容量」的位运算性质，搬迁更清晰。

### put 路径（机制）

1. 计算 hash（扰动，减少高位无用）  
2. 算下标，桶空 → 直接放  
3. 桶头 key 相等 → 覆盖 value  
4. 否则沿链/树查找；没有则追加  
5. 链长达到阈值且表够长 → 树化  
6. 元素数超过 `threshold = 容量 × 负载因子(默认 0.75)` → 扩容为 2 倍并 rehash  

### 为什么要树化阈值 8、表长 64

- 短链时链表更简单、内存更省，常数更小  
- 泊松分布下，理想 hash 出现很长链的概率很低；8 是工程折中  
- 表还很短时优先**扩容摊平**，而不是急着为每个桶建树  

### 为什么默认线程不安全

单线程数据结构更简单更快。多线程并发 put 可能丢更新、出现数据竞争。需要并发时用 `ConcurrentHashMap`（分段/CAS+同步的演进实现），而不是多线程共用 HashMap。

### 负载因子 0.75 的权衡

- 更小：空间浪费，冲突少  
- 更大：更省空间，冲突与树化更多  
  0.75 是时间和空间的历史经验默认值。

## 五、ArrayList / 迭代

### ArrayList

底层扩容数组：随机访问 O(1)；尾部追加均摊 O(1)；中间插入要搬移。  
为何常用：缓存友好、实现简单、绝大多数业务「读多、尾部写」。

### fail-fast

用迭代器遍历时，若结构被**另一路径**修改（非迭代器自己的 remove），可能抛 `ConcurrentModificationException`。目的是尽早暴露并发/错误修改，而不是默默给错数据。并发容器的迭代语义通常是**弱一致**，不走同一套 CME 策略。

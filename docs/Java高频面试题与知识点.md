## 这个专题在问啥？

就三件事，搞懂就够用：

1. 两个对象怎么算「一样」？  
2. 字符串为啥改不了？  
3. **HashMap 怎么存、怎么找**（必问）

---

## 1. == 和 equals：到底比什么？

### 先记一句

- `==`：基本类型比**数一不一样**；对象比**是不是同一个东西**。  
- `equals`：可以改成比**内容一不一样**（比如两个用户 id 相同就算同一个人）。

### 生活例子

- 你和你的双胞胎：**两个人**（`==` 是 false），但身份证号可以做成「业务上相等」（`equals` true）。  
- 同一个水杯你左手递右手：还是**那一个杯子**（`==` true）。

### 慢慢讲

Java 里每个对象默认的 `equals`，其实也是比「是不是同一个」。  
String、Integer 这些类**重写了** equals，所以 `"hello".equals("hello")` 是 true，即使可能不是同一个对象。

### hashCode 是干嘛的？

可以想成：**快递柜编号**。  
HashMap 先看编号进哪个柜子，再在柜子里用 equals 细看是不是你的快递。

### 必须遵守的规矩（很重要）

> 如果两个对象 equals 说「一样」，那它们的 hashCode **必须一样**。

**为啥？**  
不然 HashMap 把东西放进 3 号柜，下次 equals 认为一样却去 8 号柜找——**永远找不到**。  
这不是风格问题，是会出 bug。

反过来：hashCode 一样，不一定 equals 一样（两个人可能碰巧同一柜号，再细比）。

### 易错

| 情况 | 结果 |
|------|------|
| 只改 equals 不改 hashCode | HashMap/HashSet 行为错乱 |
| 用会变的字段当 key，后来又改了 | 放进去取不出来 |

## 2. String 为啥不能改？

### 先记一句

字符串一旦创建，**里面的字就改不了**。你以为的修改，其实是**造了一个新的**。

### 生活例子

石碑刻了字（不可变）。你要改字，只能再刻一块新碑，旧的还在。  
黑板（可变）可以擦了重写——String 不是黑板。

### 为啥要设计成这样？

1. **能共用**：很多地方都写 `"OK"`，可以共用同一份，省内存。要是能改，一处改了所有人都遭殃。  
2. **当钥匙安全**：经常拿 String 当 HashMap 的 key，内容不变才能放心记住 hash。  
3. **更安全**：密码、类名、路径不会被别人半路改掉。

### 拼接怎么写？

循环里老用 `+` 会不停造新对象。  
要用 **StringBuilder**（像可擦写的草稿本，写完再变成 String）。

## 3. HashMap：最重要的一块

### 先记一句

HashMap = **一排抽屉（数组）** + 抽屉里可能挂**一条链**，链太长就改成**树**，好找东西。

### 生活例子

图书馆：

1. 先按书名算一个「架号」（hash）  
2. 走到那个书架  
3. 书架上可能有好几本，再一本本比对书名（equals）  
4. 某一格书多到爆炸，就整理成更好找的目录树（红黑树）

### 放进去（put）在干嘛？

1. 用 key 算出 hash  
2. 换算成数组下标（哪个抽屉）  
3. 抽屉空 → 直接放  
4. 不空 → 看有没有相同 key：有就**改 value**，没有就**挂到后面**  
5. 东西太多 → **扩容**（抽屉变多，重新摆）

### 为啥长度经常是 2、4、8、16…（2 的幂）？

这样可以用位运算快速算「进几号抽屉」，又快又均匀。你面试说到「2 的幂 + 位运算」就够用。

### 链多长会变树？

大概：**一条链上超过 8 个**，并且数组已经够大（≥64），才变红黑树。  
太短就急着变树不划算；太短时更可能先**扩容**把人分散到更多抽屉。

### 为啥说它线程不安全？

多个人同时往同一个 HashMap 里塞东西，可能互相覆盖、数据乱。  
多线程请用 **ConcurrentHashMap**。

### 默认 16 和 0.75 是啥？

- 一开始 16 个抽屉  
- 装到大约 75% 就扩容成两倍，免得抽屉太挤、链太长

## 4. ArrayList 一句话

底层是**可扩容数组**：按下标拿很快；中间插入要挪位置。  
日常「列表」十有八九用它。

遍历时如果一边遍历一边乱改结构，可能直接报错（fail-fast），这是故意早发现错误。

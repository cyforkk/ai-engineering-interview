## P0 长口述

### 1. Redis 为什么快

**口述：**

> 主要四点。  
> 第一，数据在**内存**，比磁盘随机读快几个数量级。  
> 第二，**单线程执行命令**，避免线程切换和锁竞争，命令处理模型简单；注意 Redis 6 的多线程主要优化网络 IO 读写，命令执行仍是单线程语义这个口径要说清。  
> 第三，底层结构高度优化，SDS、跳表、压缩列表等。  
> 第四，用 **IO 多路复用** 处理大量连接。  
> 所以它适合高并发缓存和热数据，不适合当无限大磁盘数据库硬扛全量冷数据。

---

### 2. 五大数据类型和使用场景

**口述：**

> String：最通用，缓存对象 JSON、计数器、分布式 Session、分布式锁的 key。  
> Hash：存对象多个字段，比如用户部分属性，可只改一个 field。  
> List：时间线、最新消息列表，也能做简易队列，但专业 MQ 场景一般上专用中间件。  
> Set：去重、共同关注、抽奖池。  
> ZSet：带分数的有序集合，**排行榜**最典型，也能做延时任务（score 当时间戳）。  
>
> 扩展：Bitmap 签到，HyperLogLog 估基数 UV，Stream 做消息。  
> 面试官若问 ZSet 底层：字典 + **跳表**，跳表范围查询实现简单、性能好。

---

### 3. 缓存穿透、击穿、雪崩（核心长答）

**口述：**

> 这三个名字像，但问题不一样。  
>
> **穿透**：查询一个根本不存在的数据，缓存没有，每次都打到 DB，恶意扫描不存在 id 会打穿。  
> 方案：缓存空值并设较短 TTL；布隆过滤器拦截一定不存在的 key；接口层参数校验。  
>
> **击穿**：一个**热点 key** 过期瞬间，大量并发打到 DB 重建。  
> 方案：互斥锁，只让一个线程回源重建；逻辑过期，缓存永不过期由异步线程更新；或热点 key 不过期+后台刷新。  
>
> **雪崩**：大量 key **同时**过期，或 Redis 整实例挂了，请求一起打 DB。  
> 方案：过期时间加随机值打散；Redis 高可用集群/哨兵；多级缓存；限流降级；DB 保护。  
>
> 答题时一定先定义再方案，体现你分得清。

---

### 4. Redis 分布式锁

**口述：**

> 最基本正确姿势是：  
> `SET key unique_value NX EX seconds`  
> NX 保证只有一个客户端成功；EX 防止持有者崩溃后死锁；unique_value 标识谁持有锁。  
>
> 释放锁必须用 **Lua 脚本**：先比较 value 是不是自己的，再 DEL，保证原子性。如果先 GET 再 DEL，中间可能误删别人锁。  
>
> 业务执行时间超过锁过期时间会有问题：可用 Redisson 看门狗自动续期，或把过期时间估宽。  
> 主从切换时，锁可能在新主上丢失，RedLock 有争议，很多业务接受小概率风险或配合业务 fencing token。  
> 生产我更常用 Redisson 封装，但面试要把原生 SET NX EX 原理讲清楚。

## P1 长口述

### 5. 缓存和数据库一致性

**口述：**

> 分布式里强一致成本很高，一般做**最终一致**。  
> 常用 Cache Aside：读时先缓存，没有读 DB 再回填；写时**先更新 DB，再删除缓存**，而不是先更新缓存。  
> 为什么删而不是改缓存？更新缓存容易和并发读写成脏，删了让下次读回源更简单。  
> 删缓存失败怎么办？重试、消息队列保证删除、或订阅 binlog 异步删。  
> 还有延迟双删：更新后删一次，延迟再删一次，降低并发窗口脏数据概率。  
> 要承认：极端并发仍可能短暂不一致，业务要能接受或对关键读走主库。

---

### 6. 过期删除和内存淘汰

**口述：**

> 过期 key 不是到点立刻全删。Redis 用**惰性删除**：访问时发现过期再删；加**定期删除**：抽样清理一部分。两者结合。  
> 内存到 maxmemory 后按策略淘汰，比如 allkeys-lru 在所有 key 里淘汰最近最少用；volatile-lru 只在设置了过期的 key 里淘汰；还有 LFU、随机、不淘汰直接写报错等。  
> 选型看业务：全站缓存常用 allkeys-lru。

---

### 7. RDB 和 AOF

**口述：**

> RDB 是某一时刻的内存快照，文件紧凑，恢复快，但两次快照之间的数据可能丢。fork 子进程做快照时可能有内存和延迟影响。  
> AOF 记写命令，数据更完整，文件更大，可重写压缩；fsync 策略 everysec 是常见折中。  
> 生产常混合持久化：重启用 RDB 快速打底，再用 AOF 补增量。按业务对丢失的容忍度选型。

---

### 8. 大 key 和热 key

**口述：**

> 大 key 单个 value 很大或结构元素极多，会导致阻塞、网络带宽打满、主从复制慢。要拆分、压缩、避免一次性返回巨大结构。  
> 热 key 是访问极其集中，单分片打爆。方案：本地缓存、读写分离副本、key 打散、限流。  
> 事前用 bigkeys 扫描和监控 QPS 分布。

## P2 简口述

### 9. 主从、哨兵、Cluster

**口述：**

> 主从复制提供副本和读扩展。  
> 哨兵负责监控和自动故障转移选主。  
> Cluster 把数据按 16384 槽分到多节点，水平扩展；跨槽操作受限，要注意数据倾斜和大 key。  
> 面试说清自己项目用的是哪一种。

### 10. Pipeline、事务、Lua

**口述：**

> Pipeline 批量发命令减 RTT，但**不保证原子**。  
> Redis 事务 MULTI/EXEC 和 MySQL 事务不同，没有回滚单条失败那套语义。  
> Lua 脚本在服务端原子执行，适合锁释放、库存扣减判断这类要原子的小逻辑。

---

## 场景口述

### 库存超卖

> 不能 get 再 decr 分两步。用 Lua：读库存、够就减、不够返回失败，保证原子。或 DECR 后判断负数再回滚策略，但 Lua 更直观。

### 排行榜

> ZSet，score 分数，member 用户；ZINCRBY 更新；ZREVRANGE 取 topN。

## 考前骨架

```text
快：内存+单线程命令+结构+多路复用
三连：穿透空值布隆 / 击穿互斥逻辑过期 / 雪崩打散高可用
锁：SET NX EX + unique + Lua
一致：先 DB 再删缓存
持久：RDB 快照 AOF 命令
```

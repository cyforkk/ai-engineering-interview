## 二、可变与默认参数（P0）

### 白话

- **不可变**：int、str、tuple、frozenset… 改值通常产生新对象。  
- **可变**：list、dict、set… 可以原地改。  

**坑：** `def f(a=[])` 里的 `[]` 在 **函数定义时创建一次**，所有调用默认共享同一个 list，会串数据。

**正确写法：**

```python
def f(a=None):
    if a is None:
        a = []
```

浅拷贝：`copy.copy` / 切片，只复制外层；深拷贝：`copy.deepcopy` 递归复制。

### 面试怎么答（约 45 秒）

> Python 里 list、dict 可变，str、tuple 不可变。  
> 默认参数如果是可变对象，定义时只创建一次，多次调用会共享，属于常见坑，应该用 None 再在函数体内初始化。  
> 拷贝要区分浅拷贝和深拷贝，嵌套结构用深拷贝才独立。

## 三、装饰器 / 生成器 / with（P0）

### 白话

**装饰器：** 接收函数、返回函数的高阶用法，`@dec` 等价于 `f = dec(f)`。用途：日志、鉴权、计时、重试、缓存。用 `functools.wraps` 保留原函数元数据，避免调试时名字全变成 wrapper。

**生成器：** 函数里 `yield`，调用得到迭代器，**惰性**产出。读大文件逐行、流水线处理时省内存。

**with：** 上下文管理器，保证 `__enter__`/`__exit__` 成对，异常也会清理（关文件、放锁、还连接）。

### 面试怎么答

> 装饰器用来无侵入地增强函数；生成器用 yield 做惰性序列省内存；with 用上下文协议保证资源释放。  
> 手写装饰器我会提到 wraps；手写 with 可以提 contextmanager 或类实现。

## 四、GIL 与并发选型（P0，最重要）

### 白话

CPython 有 **全球解释器锁 GIL**：同一个进程里，**同一时刻通常只有一个线程在执行 Python 字节码**。  

因此：

| 负载类型 | 更合适 | 原因 |
|----------|--------|------|
| CPU 密集（重计算） | **多进程** / 推到 C 扩展 / 外部服务 | 多线程跑不满多核 |
| IO 密集（等网络、磁盘） | **多线程** 或 **asyncio** | 等待时会释放 GIL 或让出事件循环 |
| 高并发等 LLM/HTTP | **asyncio** + 异步客户端 | 单线程撑大量等待连接 |

**asyncio 大忌：** 在协程里写 `time.sleep`、同步 `requests`、重 CPU 循环 → **堵住事件循环**，所有请求一起卡。应使用 `asyncio.sleep`、httpx/aiohttp 异步库，CPU 重活丢进程池。

### 面试怎么答（约 1.5～2 分钟，可照着说）

> CPython 的 GIL 导致多线程做纯 Python CPU 计算很难吃满多核，CPU 密集我会用多进程或把计算卸载出去。  
> IO 密集时线程仍有价值，因为等待 IO 会释放 GIL。  
> 做 AI 网关、高并发调模型 API 时我优先 asyncio：用协程在单线程里交错等待大量网络 IO。  
> 但必须保证协程路径上没有阻塞调用，否则事件循环被堵，延迟会雪崩。  
> FastAPI 很适合这类服务：类型注解、async 路由、自动 OpenAPI；还要自己做超时、限流、连接池和观测。

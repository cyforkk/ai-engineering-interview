## 一、Agent 相对 Chat 多了什么

Chat：上下文 → 文本。  
Agent：目标 → **多步决策** → **调用工具**改变世界或获取信息 → 观察 → 再决策。  

公式：`Agent ≈ LLM + 规划 + 记忆 + 工具 + 停止条件/护栏`。

---

## 二、为什么不能处处 Agent

| 方式 | 优点 | 代价 |
|------|------|------|
| 工作流 | 可控、可测、便宜 | 僵硬 |
| Agent | 灵活、少硬编码分支 | 贵、慢、难测、易循环 |

合规、固定 SOP、成本敏感 → 工作流（或工作流套一点工具）。  
探索性任务、工具组合事先写不全 → 才值得 Agent。

---

## 三、ReAct 为什么有效，又为什么危险

交错「推理与行动」让模型基于真实观察调整，而不是一次幻觉到底。  

危险在于：模型可能重复无效动作、步数膨胀、工具失败后胡乱尝试。  
故 **max_steps、去重、超时、熔断、HITL** 不是附加题，是产品能否上线的必要条件。

---

## 四、工具调用：为什么执行权必须在服务端

模型只生成「我想调 X(参数)」的结构化意图。若让模型「自己说有权限就执行」，等于把授权交给不可信生成。  
服务端：鉴权、参数白名单、超时、审计、写操作审批——这是安全边界。

---

## 五、记忆为什么要分层

窗口有限且贵。短时靠对话；长时靠外部存储检索；中间状态要压缩。  
全量历史硬塞 = 贵、慢、噪声大、还可能串隐私。

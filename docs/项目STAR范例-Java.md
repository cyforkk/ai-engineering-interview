## 范例：订单查询与缓存治理

### 一分钟介绍

> 我参与的是电商交易里的订单查询与详情服务，给 App 和客服后台用。  
> 技术栈 JDK（17）+ Spring Boot，MySQL，Redis，MQ 用（RocketMQ/Kafka）。  
> 峰值查询 QPS 大约（2k～4k）。我主要负责详情读链路、缓存与一次 Full GC/超时治理。  
> 优化后核心接口 P99 从约（800ms）降到（120ms），Full GC 从高峰频繁变为偶发。

---

### 难点 STAR（缓存 + SQL）

**S：** 大促前详情接口变慢，DB CPU 高，缓存命中率只有（40%）左右。  

**T：** 大促前把 P99 压到（200ms）内，DB 可顶住峰值。  

**A：**  
1. 慢 SQL：explain 发现未走组合索引，补（user_id, create_time）并去掉 select *。  
2. 缓存：Cache Aside，key 按订单号；空值短 TTL 防穿透；过期加随机防雪崩；热点互斥重建。  
3. 本地缓存（Caffeine）挡最热店铺配置，设 maximumSize。  
4. 线程池：下游调用隔离，避免慢 HTTP 占满 Tomcat 线程。  
5. 观测：命中率、DB QPS、接口 P99 大盘。  

**R：** P99（800→120ms）；命中率（40%→85%+）；大促 DB CPU 安全水位。  

**复盘：** 先 explain 再堆缓存；本地缓存必须有上限。

---

### 故障 STAR（Full GC / 超时）

**现象：** 晚高峰周期性超时，与 GC 日志 pause 对齐，Old 区升高。  

**排查：** jstat 见 Full GC 后回收差；dump 发现某静态 Map 做本地缓存无淘汰，堆被查询结果撑满。  

**止血：** 重启+临时关该缓存开关；扩容顶峰。  

**根治：** 换成 Caffeine 有界+过期；禁止无界静态缓存 code review 规则；HeapDumpOnOOM。  

**结果：** Full GC 恢复正常；P99 稳定。

### 可追问预备

| 问 | 答要点 |
|----|--------|
| 一致性？ | 先更 DB 再删缓存；接受短暂不一致 |

| 线程池参数？ | 有界队列+拒绝；按 IO 密压测 |

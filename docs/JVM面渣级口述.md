## 一、P0 核心概念口述

### 1. 说一下 JVM 内存区域

**口述：**

> JVM 运行时数据区主要分五块，再加上一块常考的直接内存。  
>
> **线程私有**的有三个：  
> 一是**程序计数器**，记当前线程执行到哪条字节码，线程切换后靠它恢复；执行 native 时是 undefined，规范里它是唯一不规定 OOM 的区域。  
> 二是**虚拟机栈**，每个方法对应一个栈帧，里面有局部变量表、操作数栈、动态链接、方法出口；栈太深会 StackOverflow，扩展失败可能 OOM。  
> 三是**本地方法栈**，给 native 方法用，HotSpot 里常和虚拟机栈一起实现。  
>
> **线程共享**的有两个：  
> **堆**最大，放对象实例和数组，是 GC 主战场，分新生代、老年代。  
> **方法区**是规范概念，JDK8 用**元空间**实现，放类元数据、运行时常量等，用本地内存，会 Metaspace OOM。  
>
> 另外 **直接内存**不属于运行时数据区，NIO 会用，也可能 Direct buffer memory OOM。  
>
> （加分）JDK7 字符串常量池进堆，JDK8 永久代被元空间替换，是为了少 PermGen OOM、用本地内存更灵活。

**30 秒精简版：**  
> 私有：计数器、栈、本地方法栈；共享：堆、方法区/元空间；还有堆外直接内存。

---

### 2. 堆和栈有什么区别

**口述：**

> 栈是线程私有的，主要管方法调用：局部变量、引用、操作数都在栈帧里，方法结束帧弹出，生命周期跟方法绑定，一般不靠 GC 回收。  
> 堆是线程共享的，对象实例在这分配，生命周期由 GC 决定，空间大但分配和回收更重。  
> 栈上存的是基本类型值和对象引用，对象本体在堆上——当然逃逸分析后也可能栈上分配，那是优化，不是默认模型。  
> 栈溢出常见是递归过深；堆溢出常见是对象太多或泄漏。

---

### 3. 如何判断对象能否被回收

**口述：**

> 主流是**可达性分析**：从 GC Roots 出发沿着引用链走，走不到的对象就是可回收的。  
> Roots 包括：虚拟机栈里的引用、静态变量、常量、JNI 引用、被同步锁持有的对象等。  
> 引用计数法给对象数引用，为 0 就回收，但**解决不了循环引用**，JVM 不用它做主判定。  
> 另外还有强软弱虚四种引用，强度不同，回收时机不同：软引用内存不够收，弱引用下次 GC 就收。

---

### 4. 垃圾回收算法有哪些

**口述：**

> 三种基础算法加分代：  
> **标记-清除**：先标后清，实现简单，但碎片多，以后大对象可能分不出连续空间。  
> **复制**：活对象拷到另一半，无碎片，适合新生代这种存活少的；代价是要预留空间，经典 Eden+两个 Survivor。  
> **标记-整理**：标完把活对象往一端挪再清边界外，无碎片，适合老年代存活多、复制成本高的场景。  
> **分代收集**是工程策略：新生代用复制，老年代用清除或整理，组合效率最高。

---

### 5. Minor GC 和 Full GC

**口述：**

> Minor 或 Young GC 只收新生代，一般 Eden 满就触发，频率高、停顿相对短。  
> Full GC 范围大，通常整堆，还可能带元空间等，STW 更长，线上要尽量少。  
> 注意不同收集器名词不完全统一：G1 还有 Mixed GC，是年轻代加部分老 Region，不等于 Full。  
> Full 常见诱因：老年代不够、元空间不够、晋升失败、显式 System.gc、并发模式失败（CMS 历史）等。

---

### 6. 讲讲 G1（可对比 CMS）

**口述：**

> G1 把堆切成很多 Region，逻辑上仍有年轻/老，但回收时按「收益」优先收垃圾多的 Region，所以叫 Garbage First。  
> 目标是在吞吐和停顿之间平衡，适合大堆，可设 MaxGCPauseMillis 这类停顿目标。  
> 它有并发标记，Young 和 Mixed 用复制 Evacuation，失败才会退化到 Full GC，应尽量避免。  
>
> CMS 是老年代低延迟收集器，并发标记清除，但有碎片和 concurrent mode failure，JDK9 废弃、14 移除。  
> 现在服务端默认多是 G1，要极致低延迟再看 ZGC，得结合 JDK 版本和压测。

---

### 7. 类加载过程

**口述：**

> 一个类从进内存到卸出，大致是：加载、验证、准备、解析、初始化，然后使用，最后卸载。  
> **加载**：按全名拿字节流，转成方法区结构，生成 Class 对象。  
> **验证**：保证字节码合法安全。  
> **准备**：给静态变量分配内存并设**零值**，注意 final 编译期常量另说。  
> **解析**：符号引用变直接引用。  
> **初始化**：执行 `<clinit>`，也就是静态赋值和静态块，父类优先。  
> 初始化触发是「主动使用」，比如 new、访问静态、反射、主类启动等。

---

### 8. 双亲委派

**口述：**

> 双亲委派是：类加载器收到请求，先让父加载器尝试，父加载不了自己才加载。  
> 好处主要两个：一是**保证核心类唯一**，你写个 java.lang.String 也替换不了 JDK 的；二是安全，避免核心 API 被篡改。  
> 破坏的情况：重写 loadClass、SPI 用线程上下文类加载器、Tomcat 要应用隔离、热部署自定义 ClassLoader。  
> 注意「双亲」不是继承意义上的父母，是委派层级。

---

### 9. 常见 OOM 有哪些、怎么查

**口述：**

> 先看异常类型，别一上来就调 -Xmx。  
> **Java heap space**：堆不够或泄漏——要 dump，用 MAT 看 Dominator 和 GC Roots 路径。  
> **Metaspace**：类太多或 ClassLoader 泄漏，动态代理、热部署要小心。  
> **Direct buffer memory**：堆外，NIO/Netty。  
> **unable to create new native thread**：线程过多。  
> StackOverflow 是栈太深，通常不算 OOM。  
>
> 排查套路：确认现象和是否发版/流量相关 → GC 曲线看回收后是否下降 → dump 分析 → 修代码或调容量 → 用指标验证。  
> 启动参数建议开 HeapDumpOnOutOfMemoryError。生产高峰慎用 jmap live，可能长时间 STW。

## 二、场景题面渣级口述（TOP15）

### 场景 1：线上堆 OOM

> 第一步看完整异常是不是 Java heap space，以及哪个实例、是否刚发版。  
> 第二步拿现场：最好事先配了 OOM 时自动 dump；没有就在低峰对问题实例 dump。  
> 第三步看 GC：OOM 前是否频繁 Full GC、回收后老年代是否几乎不降。  
> 第四步 MAT：谁占内存最大、引用链是谁——静态集合、无界缓存、一次加载大数据很常见。  
> 第五步定性：泄漏就修引用生命周期；就是业务要那么大内存就扩容或改成分页流式。  
> 最后验证：错误率、堆使用、Full GC 次数是否恢复。止血可以摘流重启，但必须保留分析材料。

### 场景 2：内存泄漏怎么定位

> 泄漏特征是**不该活的对象还被引用**，Full GC 后老年代降不下来，曲线缓慢抬升最后 OOM。  
> 做法：对比两次 histo 或两次 dump，看哪类实例只增不减；MAT Path to GC Roots 找到持有者。  
> 典型例子：静态 Map 当缓存无上限；线程池里 ThreadLocal 不 remove，value 挂在线程上。  
> 修：有界缓存+过期；finally remove；取消监听。验证看 Old 区是否走平。

### 场景 3：频繁 Full GC

> 先看 GC 日志频率、停顿、原因。再用 jstat 看 Full 之后 Old/Metaspace 降不降。  
> 能降：可能堆偏小或流量峰值，评估扩容和减分配。  
> 不降：高度怀疑泄漏或长生命周期对象过多，必须 dump。  
> 也要查 System.gc、元空间、大对象晋升。处理完用 Full GC 次数和 P99 验证，别只改 -Xmx 了事。

### 场景 4：频繁 Young GC

> Young GC 本身正常，过于频繁会占 CPU、加速晋升。  
> 常见原因：Eden 过小、分配速率太高——循环 new、大 JSON、字符串拼接、自动装箱。  
> 先从代码减分配，再考虑调新生代；Young 调太大也可能单次停顿变长，要结合停顿目标。

### 场景 5：CPU 100%

> top 找 Java 进程 → top -Hp 找线程 → tid 转十六进制 → jstack 对 nid。  
> 栈在业务代码：死循环、正则回溯、重计算。  
> 栈在 GC 线程：转 GC/内存问题。  
> 短时 Compiler 线程：预热可能正常。  
> Arthas thread 也能快速列出忙线程。修好后看 CPU 和 RT。

### 场景 6：卡顿是不是 GC

> 关键是**时间对齐**：APM 上 RT 尖刺时间和 GC 日志 pause 是否重合。  
> 重合且 pause 到几百毫秒以上，优先 GC。  
> 同时排除下游慢、锁竞争、CPU steal。确认后再按 Full/Young 线深挖，不是盲目扩容实例。

### 场景 7：堆外 / DirectMemory

> 表现是进程 RSS 很高但 Java 堆监控不高，或直接 Direct buffer memory。  
> 查 NIO/Netty、MaxDirectMemorySize、是否未释放。  
> 处理：规范释放、池化参数、限制上限、容器 limit 要算上堆外。

### 场景 8：Metaspace 持续上涨

> 不是业务对象泄漏，是类或 ClassLoader 泄漏。  
> 动态代理、脚本、热部署反复加载最常见。  
> 查已加载类数量是否只增不减；设 MaxMetaspaceSize 防无限吃内存，但根治要修加载器泄漏。

### 场景 9：线程死锁

> 部分请求卡住、CPU 不一定高。jstack 常直接报 Found deadlock，给出锁图。  
> Arthas thread -b 找阻塞。  
> 止血重启；根治统一锁顺序、缩小锁范围、tryLock。压测验证不再卡死。

### 场景 10：容器 OOMKilled 无 Java OOM

> 被 cgroup 杀掉，RSS 超 limit，不一定走到堆 OOM。  
> -Xmx 贴满 limit、堆外、元空间、线程栈都占内存；老 JDK 可能不感知容器。  
> 堆用 MaxRAMPercentage 留 25%+ 给非堆；限线程和 Direct；升级容器感知 JDK；监控容器内存不只堆。

### 场景 11：大促前评估

> 压测峰值看存活集、GC 频率、P99。  
> 堆明显大于存活集并留峰值余量；Xms=Xmx 减动态扩容抖动。  
> 配 GC 日志和 HeapDump；选 G1/ZGC 看 SLA；预案限流扩容；盯 GC 和错误率告警。

### 场景 12：CMS 迁 G1 / 上 ZGC

> 不是改一个参数就完。要压测停顿、吞吐、CPU、晋升失败是否消失。  
> G1 看 Mixed 和停顿目标是否过激。ZGC 看版本和内存/CPU 开销，灰度对比 P99 再全量。为 SLA 服务不跟风。

### 场景 13：dump 太大 / 线上 jmap

> MAT 要机器内存够；可先 histo 缩小范围。  
> 线上高峰不做 live dump；优先自动 dump 和轻量指标；必须 dump 就摘流或低峰。

### 场景 14：线程池满与 JVM

> 有界队列满触发拒绝；队列堆积任务对象占堆；无线程上限可能线程 OOM。  
> 和无界队列任务无限堆导致堆 OOM 是一类问题。要有界+拒绝+隔离+监控。

### 场景 15：讲一次真实经历（STAR 模板）

> **S：** 晚高峰 P99 从 80ms 到 1s+，内存爬升。  
> **T：** 当天止血并找根因。  
> **A：** GC 与超时时间对齐 → jstat → dump → 静态缓存无上限 → 紧急去缓存/加上限 → Caffeine 有界+过期+告警。  
> **R：** Full GC 恢复，P99 回落；复盘同类缓存。  
> 强调工具名和前后数据，别只说「优化了」。

## 三、P1 追问短答（面渣补刀）

| 追问 | 答 |
|------|-----|
| 对象一定在堆？ | 不一定，逃逸分析可栈上/标量替换 |
| 软引用弱引用？ | 软：内存紧收；弱：下次 GC 收；ThreadLocal key 是弱引用 |
| 晋升条件？ | 年龄、大对象、动态年龄、Survivor 放不下 |
| STW？ | 停用户线程保一致性；在安全点暂停 |
| 双亲委派如何破坏？ | 重写 loadClass、TCCL、Tomcat |
| jmap 会 STW 吗？ | live 可能触发 Full GC，生产慎用 |

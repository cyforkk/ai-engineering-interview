## 范例：企业知识库问答（RAG）

### 一分钟介绍

> 我做的是内部「制度与产品文档问答」助手，给员工在飞书/Web 里问制度、产品说明。  
> 技术栈 Python + FastAPI，向量检索用 pgvector，模型走云 API，前面网关鉴权限流。  
> 文档量大约（2 万）篇 chunk，日均问答（3k～5k）次。我负责检索链路、权限过滤、评测集和接口稳定性。  
> 上线后人工抽检满意率大约从（55%）到（82%），P99 控制在（3.5s）内。

---

### 难点 STAR

**S：** 早期纯提示词塞全文，又贵又慢，还经常串部门文档。  

**T：** 三个月内做成可权限隔离的 RAG，核心问法准确率可评测，延迟可接受。  

**A：**  
1. 解析与分块：按标题切，重叠（100～200）token，表格单独策略。  
2. 检索：embedding + 元数据 `tenant/dept` 强制 filter；加关键词混合补专有名词。  
3. 生成：低温 + 强制引用 chunk_id；低分拒答。  
4. 工程：httpx 异步、Semaphore 限制模型并发、超时与有限重试；Prompt/分块策略版本化。  
5. 评测：金标（200）条，看命中与忠实度，发版回归。  

**R：** 抽检满意率（55%→82%）；拒答率可控，减少胡编；跨部门误召回抽检为 0；单次成本下降约（40%）因为不再塞长文。  

**复盘：** 分块质量比换更大模型更重要；权限 filter 必须服务端硬编码。

---

### 故障 STAR

**现象：** 某版本上线后超时率升高，P99 到（10s+）。  

**排查：** 分阶段日志发现重排模型同步调用堵了事件循环；同时 topK 从 5 调到 20。  

**止血：** 回滚 topK；重排改线程池/缩小候选。  

**根治：** async 化或独立重排服务；默认 topK 配置门禁；压测基线入 CI。  

**结果：** P99 回到（3s）级；增加「上下文组装耗时」大盘。

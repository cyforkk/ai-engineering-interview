## 项目名称：企业内部知识库问答助手（RAG + 工具）

| 字段 | 填写示例 |
|------|----------|
| 业务一句话 | 让业务同学用自然语言查制度/产品文档，并支持查工单状态（工具），减少人肉翻文档 |
| 你的角色 | AI 应用开发，负责 RAG 链路、工具接入、评测门禁与一部分 Prompt 版本管理 |
| 规模数据 | 文档约 **8 万 chunk**；日均对话 **3k**；峰值并发约 50；单次目标 P95 &lt; 8s |
| 技术栈 | Python、FastAPI、向量库（pgvector/Milvus 其一）、Embedding、LLM API、Redis、LangGraph/自研状态机 |
| 最大难点 1 | 幻觉引用 + 工具乱调导致「答错还很自信」 |
| 方案与取舍 | 混合检索 + 重排 + **强制引用/拒答**；工具走白名单 Schema + 权限；**不**上完全自主 Multi-Agent（难测、贵）；高风险操作 HITL |
| 结果 | 金标集「可引用正确率」从 **62%→86%**；工具调用正确率 **91%**；严重幻觉工单下降约 **70%** |
| 复盘 | 切片策略初期一刀切 500 字，表格/制度条款应结构化；评测集要持续扩难例，不能只刷简单题 |

### 可挂靠的高频场景

| 场景 | 挂靠 | 项目里对应什么 | 2 分钟口述要点 |
|------|:----:|----------------|----------------|
| RAG / 检索 | ☑ | 主链路 | ① 解析切块重叠 ② 向量+关键词 ③ 重排 ④ 带引用生成 ⑤ 无依据拒答 |
| Agent / 工具 | ☑ | 查工单/搜文档 | ① ReAct 限 max_steps ② 工具 JSON Schema ③ 服务端鉴权 ④ 失败熔断 |
| 评测 / LLMOps | ☑ | 发版门禁 | ① 金标 200 条 ② 正确率/引用率/费用 ③ Prompt 版本 ④ 不达标不发 |
| 安全 / 注入 | ☑ | 文档间接注入 | ① 系统/用户/工具分通道 ② 工具最小权限 ③ 输出过滤 ④ 审计 |
| 高并发 / 限流 | ☑ | 网关与模型 | ① 用户级限流 ② 模型队列 ③ 超时与降级小模型 |
| 缓存 | ☑ | 同问缓存 | ① 语义/精确缓存 ② 文档版本失效 ③ 注意隐私不乱共享 |
| 幂等 | ☑ | 工具写操作 | ① 写工具少 ② 业务单号 ③ 重试只读优先 |

### STAR 口述稿（约 3 分钟）

- **S：** 业务知识散落在 wiki/PDF，客服与运营重复答同类问题，且模型直接答容易瞎编制度条款。  
- **T：** 我负责从 0 到 1 上线可引用的知识问答，并接「查工单」一个只读工具，要求可评测、可回滚。  
- **A：** ① 建 RAG：解析→切块→embedding→混合检索→重排→生成，答案必须带 chunk 引用，否则拒答。② 工具：OpenAPI 式 Schema，服务端校验租户与权限，禁止模型直连内网任意 URL。③ 编排：默认 Workflow，仅工具选择用有限 ReAct，max_steps=6，重复动作熔断。④ 评测：金标集覆盖事实题/拒答题/工具题，CI 门禁；线上看引用点击、点踩、费用。  
- **R：** 正确率与引用率达标后全量；幻觉工单明显下降；形成 Prompt/索引版本发布流程。  

### 面试官可能追问（示例答）

1. **如何降低幻觉？**  
   检索 grounding、强制引用、低分拒答、难例进金标、定期抽检；不幻想单靠 Prompt 消除。  
2. **工具失败怎么办？**  
   超时重试（幂等读）、熔断、降级为「仅文档回答」或转人工；轨迹落日志。  
3. **为什么不全用 Agent？**  
   步骤固定的走工作流更稳更便宜；动态部分再给模型有限选择，可控性优先。

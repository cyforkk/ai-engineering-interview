# 推理部署入门 · 面渣级口述

> 配套：[推理部署高频面试题与知识点.md](./推理部署高频面试题与知识点.md)

---

### 1. 为什么推理要独立服务

**口述：**

> 大模型占显存、启动慢、扩缩容节奏和业务 API 不同。  
> 放在 Web 进程里会导致内存翻倍、发布笨重、一个慢推理拖死全部接口。  
> 正确是业务服务异步或 HTTP 调推理集群，中间可加队列和网关限流。

---

### 2. 延迟和吞吐怎么权衡

**口述：**

> 单条请求立刻算延迟最低，但 GPU 利用率差。  
> 组 batch 提高吞吐，可能让单请求多等一会儿。  
> 在线要控 P99，离线报表可加大 batch。  
> 引擎的连续批处理就是在动态做这个权衡。

---

### 3. 量化

**口述：**

> 把权重从 FP16 等到更低比特，显存更小，同卡并发更高，通常更快，但可能掉点。  
> 上线前用业务评测集看是否可接受，不要只看公开榜。

---

### 4. 显存 OOM

**口述：**

> 常见：并发太高 KV cache 爆、上下文太长、batch 太大、同卡塞多个大模型。  
> 处理：降并发、限 max tokens、量化、扩卡、隔离 embedding 与生成服务。

---

### 5. 和 API 厂商比

**口述：**

> 自建可控数据与单价（量大时），但有工程和人力成本。  
> 流量不稳或要最快迭代，云 API 更合适。  
> 可网关统一，后端切自建或云。

---

## 考前骨架

```text
业务与推理分离
吞吐靠 batch；在线控 P99
量化换显存与并发
OOM：KV/并发/长度
滚动更新+健康检查
```

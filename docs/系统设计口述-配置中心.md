# 系统设计口述 · 配置中心

> 配套：[微服务与分布式](./微服务与分布式高频面试题与知识点.md) · [后端系统设计](./后端系统设计高频面试题与知识点.md)  
> 时长：**4～6 分钟**

---

## 开场

**口述：**

> 配置中心解决：配置从代码/机器文件里拆出来，集中管理、分环境、动态推送、权限审计。  
> 假设几百～上千服务实例，配置项万级；要高可用，推送秒级生效；错误配置可回滚。

---

## 核心能力

**口述：**

> 1）按 `app/env/namespace` 隔离配置；  
> 2）发布：灰度→全量；版本号与历史；  
> 3）客户端监听变更（长轮询/websocket/挂载）；  
> 4）权限：谁改谁审；密钥类加密存；  
> 5）容灾：本地快照，中心挂了仍用上次配置启动。

---

## 架构

**口述：**

> 控制台（UI）→ 配置服务 API → 存储（MySQL/ etcd/ 自研）；  
> Server 集群无状态或选主；  
> 客户端 SDK 启动拉取 + 长轮询；变更时回调业务 `@RefreshScope` 或自建 listener。  
> 推送模型：Server 记下客户端订阅关系，配置变更时通知；或客户端短周期 pull。  
> 类似 Nacos/Apollo/Spring Cloud Config 的分工要能对上：注册与配置可一体可分离。

---

## 一致性与推送

**口述：**

> 强一致改配置：写库成功再通知。  
> 实例间生效有时间差，属最终一致；敏感开关要灰度百分比发布。  
> 推送失败靠客户端超时重新 pull 兜底。  
> 大 value 注意带宽；可拆分 dataId。

---

## 安全与治理

**口述：**

> 生产配置变更要审批；操作审计日志。  
> Secret 不进明文 Git；与 KMS 集成。  
> 防把错误配置全量推死集群：金丝雀、一键回滚、重要项校验 schema。

---

## 客户端设计

**口述：**

> 启动：拉全量 → 落本地文件 → 再进内存。  
> 运行：监听；失败重试；线程安全更新。  
> 业务：动态刷新数据源要谨慎（连接池），很多项重启才安全——要在文档标 `refreshable`。

## 追问快答

| 问 | 答 |
|----|-----|
| 和本地 yml？ | 中心管动态与多环境；默认值仍可本地 |

app/env 隔离 + 版本灰度

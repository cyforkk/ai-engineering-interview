## 二、P0 必会

### 2.1 向量库解决什么

| 点 | 结论 |
|----|------|
| 场景 | 语义检索、RAG、推荐、去重、以图搜图 |
| 输入 | embedding 向量（模型产出） |
| 查询 | 近邻 topK，非精确 SQL 等值 |
| 与 ES | ES 强词法/聚合；向量库强语义相似度；常**混合** |

### 2.2 相似度度量

| 度量 | 注意 |
|------|------|
| 余弦 cosine | 方向，常用；注意是否已归一化 |
| 内积 IP | 与归一化向量相关 |
| 欧氏 L2 | 绝对距离 |

**模型与度量要匹配**，别混用训练设定。

### 2.3 ANN vs 精确

| 点 | 结论 |
|----|------|
| Flat/精确 | 暴力算，准但慢，小数据 |
| ANN | 近似最近邻，换速度/规模 |
| 权衡 | recall、延迟、内存、构建时间 |

### 2.4 元数据过滤（企业必问）

- 按 tenant_id、权限、时间、类型 filter 再检索或检索后过滤  
- **预过滤 / 后过滤**影响性能与正确性  
- 无权限文档不能进上下文（安全）  

### 2.5 RAG 中的位置

```text
文档 → 分块 → embedding → 写入向量库
Query → embedding → 检索 topK (+ filter) → 重排 → LLM
```

## 三、P1：索引概念（口述级）

| 类型 | 直观理解 | 特点 |
|------|----------|------|
| Flat | 全比 | 准、慢 |
| IVF | 先聚类再比子集 | 省算力，要调 nprobe |
| HNSW | 多层图导航 | 高召回低延迟，吃内存 |
| 量化 PQ 等 | 压缩向量 | 省内存，略损精度 |

不要求手推公式，要求：**知道有召回-延迟-内存三角，会按数据量选型。**

## 四、Milvus（概念高频）

| 概念 | 说明 |
|------|------|
| 定位 | 开源向量数据库，面向规模化检索 |
| Collection | 类似表，含向量字段+标量字段 |
| Partition / Partition Key | 隔离与加速过滤（租户、业务线） |
| Segment / 索引构建 | 数据分段；索引构建影响时效 |
| 一致性 | 写入可见性与业务要求（最终/强）按版本与配置 |
| 部署 | Standalone / 集群；依赖对象存储与 etcd 等（知架构分层） |
| 适用 | 亿级向量、独立检索服务、多语言 SDK |

**高频题：** Collection 怎么设计？如何多租户？索引选 HNSW 还是 IVF？

## 五、pgvector（概念高频）

| 概念 | 说明 |
|------|------|
| 定位 | PostgreSQL 扩展，向量进关系库 |
| 优势 | 与业务表事务、JOIN、运维统一；上手快 |
| 索引 | ivfflat、hnsw（版本相关） |
| 适用 | 中小规模、已有 PG、强事务与 SQL 过滤 |
| 瓶颈 | 超大规模、极致 QPS 时不如专用向量库 |

**高频题：** 为什么选 pgvector 而不是 Milvus？何时迁出？

## 六、选型对比（面试表）

| 维度 | pgvector | Milvus 类专用库 |
|------|----------|-----------------|
| 规模 | 小到中 | 中到极大 |
| 运维 | 会 PG 就行 | 组件更多 |
| 事务/关联 | 强 | 弱于关系库 |
| 极致检索性能 | 一般 | 强 |
| 混合查询 | SQL 自然 | 标量字段+表达式 |
| 典型 | 创业/中小 RAG | 平台化检索 |

其它名字：Qdrant、Weaviate、Elasticsearch vector、云厂商托管——**简历写哪个深挖哪个**。

## 七、工程要点

| 点 | 实践 |
|----|------|
| 维度 | 与 embedding 模型一致 |
| 更新删除 | 软删+过滤；重建索引策略 |
| 批量写入 | 批量 embedding + upsert |
| 版本 | 模型升级要**全量重嵌** |
| 观测 | 检索延迟、空结果率、filter 命中 |
| 混合检索 | 向量 + BM25/关键词 融合 |

## 一、为什么需要向量检索

关键词检索依赖字面匹配：用户说「怎么退货」，文档写「售后流程」可能搜不到。  
Embedding 把文本映射到高维向量，**语义相近 ⇒ 向量距离近**，从而用近邻搜索找到相关段落。

这不是魔法：同义、改写、多语言在训练目标下被拉近；专有名词、极短 query、数据噪声仍会失手，所以要工程兜底。

---

## 二、索引：为何常用 ANN 而不是精确暴力

暴力对每个向量算距离：数据量×维度上来后延迟不可接受。  
ANN（近似最近邻）如 **HNSW**（分层导航小世界图）：在可接受的召回损失下换数量级加速。  
业务要调：`topK`、ef 搜索宽度、是否重排（用更强模型精排）。

---

## 三、Chunk（切块）的所以然

1. **窗口预算**：检索片段要塞进 LLM 上下文  
2. **噪声控制**：整章塞入会稀释主题，模型抓不住重点  
3. **粒度权衡**：块过大 → 嘈杂；过小 → 缺上下文、实体被切断  

实践：按标题/段落/语义切；适当重叠（overlap）减少边界切断；元数据带文档名、章节便于引用。

---

## 四、混合检索为什么常常更好

| 通路 | 擅长 |
|------|------|
| 向量 | 同义、语义 |
| 关键词（BM25 等） | 订单号、错误码、专有名词精确命中 |

融合（加权/RRF）后精排，能同时覆盖「说人话」和「搜编号」。

---

## 五、召回差时的排查顺序

1. 语料是否切错、是否过期  
2. Embedding 模型是否匹配语言与领域  
3. topK 是否过小；是否该上 hybrid  
4. query 改写/HyDE 是否需要  
5. 精排与阈值：低相关宁可不答（降幻觉）

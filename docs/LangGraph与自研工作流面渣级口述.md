## 开场：为什么需要图

纯 while-Agent 很灵活，但有三个工程问题：难测、难限步、难插入人工审批，失败了也难从中间恢复。  
全写死的工作流很可控，但不会动态选工具。  
LangGraph 这类方案用 **图 + 可序列化状态**，在灵活和可控之间取平衡。

---

## 核心概念（1 分钟）

- **State**：全局状态，比如 query、docs、answer、retry_count，可存盘  
- **Node**：一步，可能是调模型、检索、工具、人审  
- **Edge**：条件边，成功走 A，失败走 B，或循环  
- **Checkpoint**：断点，进程挂了能续跑  

相对线性 Chain，强在循环、分支、Human-in-the-loop 和可观测回放。

---

## 一个最小可讲的业务图（1 分钟）

节点：意图识别 → 检索 → 生成 → 校验。  
校验失败且 retry_count < 3 就回到检索；否则转人工。  
状态里带着 retry_count，这就是 **可控的多步智能**，而不是无限 ReAct 转圈。

和纯 Agent 比：步数、分支、人审点都是显式的，测试可以针对节点写。  
和死工作流比：某个节点内部仍可调 LLM 做动态决策。

---

## 自研要注意什么（1 分钟）

1. 状态 schema 要版本化，防止发版后旧状态读不出来  
2. 工具调用必须幂等、超时、重试有上限  
3. 每步 trace：输入输出摘要、耗时、token  
4. 费用熔断：max_steps、max_tool_calls  
5. 权限：写操作默认要人审或二次确认  

---

## 选型一句话

固定 SOP → 工作流；强探索 → Agent；要可测可恢复的多步智能 → Graph。  
项目里讲清：几个节点、哪里循环、哪里 HITL、如何限步。



## 专题坑（编排）

- **一上来全 Agent**：难测、贵、不可控；固定步骤先工作流。  
- **无 max_steps**：死循环烧 token。  
- **状态不落盘**：失败无法恢复/审计。  
- **工具权限放模型决定**：必须服务端白名单。

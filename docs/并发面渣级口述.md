## P0 长口述

### 1. 讲一下线程池（必背长答）

**口述：**

> 生产上创建线程我优先用线程池，核心是 ThreadPoolExecutor。  
>
> **关键参数**有：核心线程数 corePoolSize、最大线程数 maximumPoolSize、空闲线程存活 keepAliveTime、工作队列 workQueue、线程工厂 threadFactory、拒绝策略 handler。  
>
> **任务提交后的执行顺序**一定要说对：  
> 第一，如果正在跑的线程数小于 core，就新建核心线程来跑这个任务；  
> 第二，如果核心已经满了，任务进入工作队列排队；  
> 第三，如果队列也满了，且当前线程数还小于 max，就创建非核心线程；  
> 第四，如果线程数也到顶了，就执行拒绝策略。  
>
> 队列选型很重要：生产我倾向**有界队列**，比如 ArrayBlockingQueue，防止任务无限堆积把内存打爆。如果用无界的 LinkedBlockingQueue，队列很难满，maximumPoolSize 往往用不上，最后可能 OOM。  
>
> 拒绝策略常见四个：Abort 抛异常最常用也最明确；CallerRuns 让提交任务的线程自己跑，有背压效果；Discard 和 DiscardOldest 会丢任务，要用要非常清楚业务能否丢。  
>
> 为什么不直接用 Executors？newFixedThreadPool 默认无界队列，容易堆积；newCachedThreadPool 最大线程很大，可能创建过多线程。阿里规范也建议自己 new ThreadPoolExecutor。  
>
> 线程数经验：CPU 密集大约核数+1；IO 密集可以更大，但最终以压测为准。还要按业务隔离池，避免核心链路被批跑任务拖死。上线要监控活跃线程、队列长度、拒绝次数。

**追问：核心线程会回收吗？**  
> 默认不会；allowCoreThreadTimeOut(true) 后可以。

**追问：shutdown 区别？**  
> shutdown 不接新任务，等已提交的跑完；shutdownNow 尝试中断，返回未执行任务列表。

### 2. synchronized 原理

**口述：**

> synchronized 可以锁方法或代码块。实例方法锁的是 this，静态方法锁的是 Class 对象，代码块可以指定对象。  
> 它是可重入的，同一个线程反复进入同一把锁会成功，靠锁计数。  
> 语义上保证：同一时刻只有一个线程进临界区（原子性），解锁前修改对下一个加锁线程可见（可见性），也有 happens-before 带来的有序性。  
> 实现层面，JVM 用对象头 Mark Word 和 monitor；历史上有偏向锁、轻量级锁到重量级锁的升级路径，不同 JDK 默认策略有变化，面试说到「会根据竞争情况膨胀到重量级锁、涉及操作系统互斥」就够用。  
> 和业务上要注意：锁粒度尽量小，避免在锁里做远程调用。

---

### 3. synchronized 和 ReentrantLock 区别

**口述：**

> 两者都能做可重入互斥。  
> synchronized 是 JVM 关键字，自动加锁释放，代码简洁，适合大部分场景。  
> ReentrantLock 是 API 锁，更灵活：可以公平锁、tryLock 超时、lockInterruptibly 可中断、多个 Condition 做精确等待通知。  
> 使用上 Lock 必须在 finally 里 unlock，否则异常路径会死锁。  
> 选型：没有高级需求用 synchronized；需要 tryLock、公平、多条件队列再用 ReentrantLock。

---

### 4. volatile 关键字

**口述：**

> volatile 主要两件事：**可见性**和**有序性**。  
> 可见性：一个线程改了 volatile 变量，马上对其他线程可见，不会一直待在自己工作内存里。  
> 有序性：禁止一些指令重排，读写时插入内存屏障。  
> 它**不能保证复合操作的原子性**，比如 i++ 是读改写三步，多线程仍会丢更新，要用 AtomicInteger 或加锁。  
> 典型场景：开关标志 `while (running)`；单例双重检查里的 instance 必须 volatile，防止指令重排让别的线程看到半初始化对象。

---

### 5. CAS 是什么？ABA？

**口述：**

> CAS 是 Compare And Swap：拿期望值去比内存里的当前值，一样才更新成新值，否则失败重试。是一种乐观并发，AtomicInteger 底层就是 CAS。  
> 优点是不需要阻塞锁，高并发下在冲突不极端时性能好。  
> 缺点：自旋失败会空转吃 CPU；只能保证一个变量；还有 **ABA 问题**——值从 A 变 B 再变回 A，CAS 以为没变。解决用带版本号的 AtomicStampedReference。  
> 对高并发计数，LongAdder 用分段减轻热点 CAS 竞争。

---

### 6. sleep 和 wait 区别

**口述：**

> sleep 是 Thread 的方法，不释放持有的锁，到点自己醒，可以在任何地方调。  
> wait 是 Object 的方法，必须在 synchronized 同步块里调，会**释放锁**，让别人能进同步块；需要别人 notify/notifyAll 或超时才能继续。  
> 等待条件要用 while 判断，不要 if，防止虚假唤醒。

---

### 7. 线程有哪些状态

**口述：**

> Java 线程状态有 NEW、RUNNABLE、BLOCKED、WAITING、TIMED_WAITING、TERMINATED。  
> NEW 是 new 了还没 start；RUNNABLE 包括就绪和正在跑，JVM 不细分是否占用 CPU。  
> BLOCKED 是等 synchronized 监视器锁；WAITING 是无限等，比如 wait、join；TIMED_WAITING 是有超时的等；TERMINATED 结束。  
> 排查卡顿时 jstack 里看状态很有用：大量 BLOCKED 可能锁竞争，WAITING 可能在等条件或锁。

## P1 长口述

### 8. AQS 原理

**口述：**

> AQS 是 AbstractQueuedSynchronizer，很多同步器的基石，比如 ReentrantLock、Semaphore、CountDownLatch。  
> 核心就两样：一个 volatile 的 **state** 表示同步状态，具体含义子类定，比如重入次数或许可数；再加一个 **CLH 变体的双向队列**，抢不到资源的线程入队排队。  
> 子类实现 tryAcquire/tryRelease 等模板方法，AQS 负责排队、阻塞、唤醒。  
> 分独占和共享：独占同一时刻一个线程，如锁；共享允许多个，如信号量、门闩。  
> 面试说到「state + 队列 + CAS + 模板方法」就抓住骨架了。

---

### 9. ThreadLocal 原理和泄漏

**口述：**

> ThreadLocal 让每个线程有一份自己的变量副本，实现线程隔离，比如存用户上下文。  
> 实现上每个 Thread 有一个 ThreadLocalMap，key 是 ThreadLocal 的**弱引用**，value 是强引用。  
> 泄漏场景：线程池里线程长期复用，ThreadLocal 外部没引用了，key 被弱引用回收变成 null，但 value 还挂着，形成 Thread → Map → value 的强链路，value 收不掉。  
> 所以用完一定要在 finally 里 **remove**。不要只 set(null)。  
> 线程池场景尤其危险，这是面试必追问点。

---

### 10. 死锁

**口述：**

> 死锁是多个线程互相持有对方需要的锁，都推进不了。  
> 四个必要条件：互斥、占有且等待、不可抢占、循环等待。  
> 预防：统一加锁顺序、tryLock 超时、缩小锁范围、避免嵌套锁。  
> 排查：jstack 会直接打印 deadlock 信息；也可以看 BLOCKED 线程的锁依赖。Arthas thread -b 找最堵的线程。  
> 线上先止血重启或摘流，再改代码防再发。

---

### 11. 单例 DCL

**口述：**

> 双重检查锁定：外层 if 避免每次进同步，内层 if 保证只创建一次，中间 synchronized 保证创建互斥。  
> instance 必须 **volatile**，禁止 new 过程的指令重排，否则其他线程可能拿到没初始化完的对象。  
> 更推荐静态内部类或枚举：枚举还能防反射和序列化问题。

---

### 12. CountDownLatch / CyclicBarrier / Semaphore

**口述：**

> CountDownLatch：一个或多个线程等一组操作结束，count 减到 0 开门，**一次性**。  
> CyclicBarrier：多方互相等待到齐再继续，**可重置**再开一局，适合分阶段并行。  
> Semaphore：维护一组许可，用来限流，比如最多 N 个线程进关键段。

## 场景口述

### 接口超时、线程池打满

> 先看线程池监控：active、queue size、rejected。jstack 看线程是在跑业务、等锁还是等下游 IO。  
> 若下游慢导致占满：加超时、隔离池、熔断。  
> 若队列无限涨：改有界+拒绝策略，防 OOM。  
> 若就是流量大：扩容或限流。验证拒绝次数下降、P99 恢复。

### CPU 飙高（并发视角）

> 同样 top + jstack。看是计算热点、死循环还是锁自旋激烈。配合业务日志定位接口。

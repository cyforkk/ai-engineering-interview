# 一、超高频（几乎必问）

## 1. TCP 核心

### 1.1 TCP vs UDP？

| | TCP | UDP |
|--|-----|-----|
| 连接 | 面向连接 | 无连接 |
| 可靠 | 可靠、有序 | 尽最大努力 |
| 速度 | 相对慢 | 快 |
| 场景 | HTTP、RPC、DB | DNS、直播、实时游戏 |

### 1.2 三次握手？为何三次？

```text
客户端                服务端
  |---- SYN=1,seq=x ---->|  SYN_SENT
  |<-- SYN=1,ACK=1,seq=y,ack=x+1 --|  SYN_RCVD
  |---- ACK=1,ack=y+1 -->|  ESTABLISHED
       ESTABLISHED
```

**为何不是两次？**

- 无法确认「客户端的接收能力」  
- 旧重复 SYN 可能导致错误连接占用资源  
- 第三次 ACK 确认双方收发都正常，同步初始序号  

**为何不四次？** 三次已够同步双向；再多无必要。

### 1.3 四次挥手？TIME_WAIT？

```text
主动关闭方              被动关闭方
  |---- FIN ---->|
  |<--- ACK ----|     （被动方可能还有数据）
  |<--- FIN ----|
  |---- ACK ----|  → 主动方 TIME_WAIT (2MSL) → CLOSED
```

**为何四次？** 全双工，每个方向单独关。  
**TIME_WAIT 作用：**

1. 保证最后 ACK 丢失时能处理重传 FIN  
2. 让旧连接报文在网络中消失，不污染新连接  

**TIME_WAIT 过多？**

- 短连接过多  
- 优化：连接复用 Keep-Alive、HTTP/2、架构合并连接  
- 调参（`tcp_tw_reuse` 等）要谨慎，懂原理再改  

**CLOSE_WAIT 多？** 应用未 close，代码泄漏连接。

### 1.4 可靠传输机制？

| 机制 | 作用 |
|------|------|
| 序列号/确认号 | 有序、确认收到 |
| 超时重传 | 丢包重发 |
| 滑动窗口 | 流水线发送 |
| 流量控制 | 接收方窗口防压垮 |
| 拥塞控制 | 防压垮网络 |

### 1.5 粘包 / 拆包？

- TCP 字节流无消息边界  
- **解决**：长度前缀、分隔符、定长、应用层协议（HTTP 自带边界）  

### 1.6 拥塞控制四阶段？

```text
慢启动 → 拥塞避免 →（丢包）快重传 → 快恢复
```

| 阶段 | 直觉 |
|------|------|
| 慢启动 | cwnd 指数增 |
| 拥塞避免 | 达阈值后线性增 |
| 快重传 | 重复 ACK 立即重传 |
| 快恢复 | 减半后进入拥塞避免等 |

## 2. HTTP / HTTPS

### 2.1 HTTP vs HTTPS？

| | HTTP | HTTPS |
|--|------|-------|
| 安全 | 明文 | TLS 加密 |
| 端口 | 80 | 443 |
| 证书 | 无 | 需要 |
| 性能 | 略快 | 握手多一点（可优化） |

### 2.2 HTTPS / TLS 握手（主干）

```text
1. ClientHello（版本、套件、随机数…）
2. ServerHello + 证书
3. 客户端验证证书（CA 链）
4. 密钥协商（非对称/ECDHE 等）→ 得到会话密钥
5. 之后用对称加密传应用数据
```

- **为何混合**：非对称换钥安全；对称加密传数据快  
- 证书：证明服务器身份，防中间人  

### 2.3 常见状态码？

| 码 | 含义 |
|----|------|
| 200 | 成功 |
| 301 | 永久重定向 |
| 302 | 临时重定向 |
| 304 | 未修改（缓存） |
| 400 | 请求错误 |
| **401** | 未认证 |
| **403** | 无权限 |
| 404 | 不存在 |
| 500 | 服务器内部错误 |
| **502** | 网关坏上游 |
| **504** | 网关等上游超时 |

### 2.4 GET vs POST？

| | GET | POST |
|--|-----|------|
| 语义 | 获取 | 提交/变更 |
| 参数 | URL/查询串 | Body |
| 幂等 | 通常是 | 不一定 |
| 缓存 | 易缓存 | 一般不 |

- 本质都是请求；安全与幂等是语义约定，不是加密。  

### 2.5 无状态如何保持？

- Cookie / Session / Token（JWT 等）  
- 服务端会话 或 无状态 Token  

### 2.6 Cookie vs Session？

| | Cookie | Session |
|--|--------|---------|
| 存哪 | 客户端 | 服务端（常见） |
| 容量 | 小 | 相对大 |
| 安全 | 可被窃 | SessionID 仍靠 Cookie 带 |

- 联系：常把 SessionID 放 Cookie  
- Token：可无服务端 Session，利于扩展

## 3. 网络模型与全流程

### 3.1 OSI 七层 vs TCP/IP 四层？

| OSI | TCP/IP 常见 | 例 |
|-----|-------------|-----|
| 应用 | 应用层 | HTTP、DNS、FTP |
| 表示/会话 | （并入应用） | — |
| 传输 | 传输层 | TCP、UDP |
| 网络 | 网际层 | IP、ICMP |
| 数据链路/物理 | 网络接口层 | 以太网、ARP |

### 3.2 从 URL 到页面？（综合必答）

```text
1. 解析 URL
2. DNS 解析得 IP（缓存 → 递归/迭代）
3. 建立 TCP（三次握手）
4. 若 HTTPS：TLS 握手
5. 发送 HTTP 请求
6. 服务器处理返回响应
7. 浏览器解析 HTML/CSS/JS，加载子资源
8. 渲染页面
（连接关闭：挥手或 Keep-Alive 复用）
```

---

## 4. DNS

### 4.1 解析过程？

```text
浏览器缓存 → 系统缓存 → 本地 hosts
 → 本地 DNS（递归）
 → 根 → 顶级域 → 权威 DNS（迭代）
 → 返回 IP
```

### 4.2 为何 UDP？何时 TCP？

- 查询短、快，用 **UDP 53**  
- **TCP**：区域传送、响应超过 512 字节截断后等

# 二、高频

## 5. TCP 进阶

### 5.1 半连接 / 全连接队列？SYN Flood？

| 队列 | 状态 |
|------|------|
| 半连接 | SYN_RCVD（未完成三次） |
| 全连接 | 已 ESTABLISHED 等 accept |

- **SYN Flood**：伪造大量 SYN 占满半连接  
- 防御：SYN Cookie、扩大队列、防火墙、限流  

### 5.2 第三次握手能否带数据？

- 可以（TFO 等场景）；标准学习以握手本身为主  

### 5.3 流量控制 vs 拥塞控制？

| | 流量控制 | 拥塞控制 |
|--|----------|----------|
| 对象 | 接收方处理能力 | 网络是否拥塞 |
| 手段 | 接收窗口 rwnd | cwnd 慢启动等 |

- 发送窗口受 min(rwnd, cwnd) 限制  

### 5.4 滑动窗口？

- 允许未确认前连续发多个段，提高吞吐  

### 5.5 Nagle 与延迟确认？

- Nagle：小包合并减小段数量  
- 延迟 ACK：合并确认  
- 两者叠加可能延迟 → 交互场景可 `TCP_NODELAY`  

---

## 6. HTTP 进阶

### 6.1 HTTP 版本演进

| 版本 | 特点 |
|------|------|
| 1.0 | 短连接为主 |
| **1.1** | Keep-Alive、管道（少用）、Host |
| **2** | 二进制帧、多路复用、头部压缩、服务器推送 |
| **3** | 基于 **QUIC（UDP）**，解决 TCP 队头阻塞 |

### 6.2 队头阻塞？

- HTTP/1.x：一连接一请求排队  
- **HTTP/2**：多路复用解决应用层队头，但 **TCP 丢包仍堵整连接**  
- **HTTP/3/QUIC**：流独立，减 TCP 层队头阻塞  

### 6.3 Keep-Alive？

- 复用连接，减握手  
- 缺点：占服务端连接资源，要超时回收  

### 6.4 301 vs 302？

| 301 | 302 |
|-----|-----|
| 永久 | 临时 |
| 缓存友好，SEO 权重转移 | 不永久转移 |

### 6.5 幂等？

- 同一请求执行多次效果相同  
- **幂等**：GET、PUT、DELETE 等（语义）  
- POST 通常非幂等

## 7. 其他高频

### WebSocket

- 全双端长连接；先 HTTP 升级握手  
- 适合 IM、实时推送  
- vs HTTP：服务端可主动推  

### 对称 vs 非对称加密

| | 对称 AES | 非对称 RSA |
|--|----------|------------|
| 密钥 | 同一把 | 公私钥 |
| 速度 | 快 | 慢 |
| HTTPS | 传数据 | 换钥/签名 |

### JWT

- Header.Payload.Signature  
- 无状态易扩展；注意密钥、过期、注销难、体积  

### ARP / ping

- **ARP**：IP → MAC  
- **ping**：ICMP 回显，测连通与延迟

| 概念 | 要点 |

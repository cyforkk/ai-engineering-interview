## 开场

Prompt 工程不是玄学咒语，而是把业务约束写成模型输入契约，并且 **可模板化、可版本、可评测**。

---

## 生产级 Prompt 我怎么写（2 分钟）

第一块 **系统角色**：你是谁、服务什么场景。  
第二块 **硬边界**：不能编造；资料不足必须说不知道或返回明确拒答结构。  
第三块 **输入槽位**：用户问题、检索文档、工具结果分开标记，避免糊成一段导致注入。  
第四块 **输出合同**：JSON 字段、枚举、错误码，方便程序解析和校验。  
第五块 **少样本**：一到三个示范，稳住格式。  
第六块 **安全**：用户内容当数据，不当系统指令。

写完进 Git，改动走评测门禁，不达标不发布。

---

## 和 RAG、Agent 的配合

RAG：明确「仅依据下列资料回答，并给出引用编号」。  
Agent：规定何时调工具、失败如何停止、最大步数。  
温度：事实与结构化任务偏低。

---

## 坏 Prompt 特征

规则互相矛盾；没有输出格式；没有拒答出口；系统和用户输入混在一起。  
好 Prompt 像接口文档：输入槽、输出 schema、异常约定。

---

## 追问

**越写越长越好吗？** 不是，要用金标驱动删减。  
**如何证明改动有效？** 对比改前改后金标指标和坏 case。  
**项目：** 举一次字段抽取准确率或拒答率提升的改动。

---

## 收尾

Prompt 是产品配置。版本、评测、回滚三者缺一不可。



## 专题坑（Prompt）

- **无版本**：线上改了无法回滚、无法对照评测。  
- **只靠自然语言约束 JSON**：要 Schema + 服务端解析校验。  
- **材料与指令混在一起**：检索文档应当数据，不当系统指令。  
- **无金标回归**：改一句 Prompt 可能静默掉点。

## P0 加厚：结构化输出与版本（冲刺必背）

**口述：**

> 生产里 Prompt 是配置不是聊天记录。我会：  
> ① 系统角色固定边界（能做什么/不能做什么）。  
> ② 要工具或 JSON 时给 **Schema 示例**，并在服务端再解析校验，失败重试或降级。  
> ③ 把检索到的材料放在明确区块，要求「仅依据材料，无则说不知道」。  
> ④ Prompt 变更走版本号 + 金标回归，和代码一样可回滚。  
> 调参：温度低更稳；创意任务再升高。别把业务规则全塞进超长 Prompt，规则能代码化就代码化。

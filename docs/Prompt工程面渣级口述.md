### 1. 好 Prompt 怎么写

**口述：**

> 我会写清角色、任务、约束和输出格式，必要时空信息怎么处理。  
> 少用「尽量专业」这种虚词，改成可检查的要求，比如「用 JSON，含 answer 和 citations 数组」。  
> 复杂任务拆成步骤，或先让模型列提纲再填，比一句话要求它又对又全更稳。

---

### 2. Few-shot 和 CoT

**口述：**

> Few-shot 用 2～5 个输入输出样例锚定格式和边界，样例要包含难例，别全是简单句。  
> 逐步思考适合推理题，但会增加 token 和延迟；对用户可见时可能泄露中间错误，有的产品用内部思考+对外简洁答案。  
> 不是所有题都开 CoT，分类抽字段往往 schema + few-shot 就够。

---

### 3. 结构化输出

**口述：**

> 业务要进系统就必须结构化。做法：Prompt 规定 schema；有的 API 支持 JSON mode；解析失败按规则重试或降级人工。  
> 校验放在代码里，不信任模型一定守格式。

---

### 4. RAG 怎么拼 Prompt

**口述：**

> 系统层写清「仅依据给定资料、无资料拒答、标注引用」。  
> 资料块带来源 id，和正文分开。  
> 用户问题单独一段，避免和资料粘在一起难区分。  
> 资料过长就重排后截断，保证最相关的在窗口前部或按模型特性放置。

---

### 5. Prompt 工程化

**口述：**

> 生产环境 Prompt 是代码资产：模板变量、版本号、与评测集绑定。  
> 改一句话都可能回退效果，所以要回归和灰度，而不是线上直接改。  
> 和产品文案一样走评审，关键合规话术冻结。

## 二、运行时内存（P0）

### 白话

程序跑起来，JVM 把内存分成几块干不同的事：

| 区域 | 人话 | 私有？ |
|------|------|--------|
| 程序计数器 | 当前线程执行到哪一行字节码 | 私有 |
| 虚拟机栈 | 方法调用的「栈帧」：局部变量、操作数 | 私有 |
| 本地方法栈 | native 方法用 | 私有 |
| **堆** | new 出来的对象，GC 主战场 | 共享 |
| 方法区/元空间 | 类的元数据等 | 共享 |
| 直接内存 | NIO 堆外，也受本机限制 | 堆外 |

**堆再分代：** 新生代（Eden + 两个 Survivor）放朝生夕死对象；老年代放活得久的。  
**JDK8：** 永久代 → **元空间**（本地内存），类太多会 Metaspace OOM。

### 面试怎么答

> 线程私有：计数器、栈、本地方法栈；共享：堆、方法区/元空间。  
> 堆放对象并分代；栈放栈帧。  
> 直接内存也要防 OOM。

### 追问

| 问 | 答 |
|----|-----|
| 栈溢出？ | 递归过深 StackOverflow；扩展失败可能 OOM |
| 对象都在堆？ | 一般是；逃逸分析可栈上分配 |
| 为何弃永久代？ | 易 OOM、大小难调；元空间用本地内存更灵活 |

## 三、垃圾回收（P0）

### 白话

**谁是垃圾：** 从 GC Roots 出发沿引用走不到的对象（可达性分析）。  
不是引用计数（解决不了循环引用）。

**Roots 常考：** 栈上引用、静态变量、常量、JNI、锁持有对象等。

**怎么收（算法）：**

| 算法 | 人话 | 典型 |
|------|------|------|
| 标记-清除 | 标完删，碎片多 | 老年代历史 |
| 复制 | 活的拷到另一半，适合活得少 | 新生代 |
| 标记-整理 | 标完挤到一边再清 | 老年代 |
| 分代 | 新区老区不同策略 | 现代基础 |

**Minor GC：** 收新生代，Eden 满常触发，相对勤、相对快。  
**Full GC：** 范围大、停顿长，要尽量少。  
**G1：** 堆切 Region，优先收「垃圾多」的区，JDK9+ 常见默认；超低延迟看 ZGC。

### 面试怎么答

> 可达性判断；新生代复制、老年代整理/清除；G1 按 Region 收益回收。  
> 关注 Young/Full 频率与停顿。

**完整稿：** [JVM面渣 · GC](./JVM面渣级口述.md)

## 四、类加载与双亲委派（P0）

### 白话

类不是用到才「整个出现」：要加载字节码、验证、准备静态变量、解析引用、初始化（执行静态块）。  
**双亲委派：** 加载请求先给父加载器，父没有再自己加载——保证 `java.lang.String` 等核心类唯一、防篡改。

### 面试怎么答

> 加载→验证→准备→解析→初始化。  
> 双亲委派先父后子；可破坏（SPI、Tomcat 隔离、自定义 ClassLoader）。

**完整稿：** [JVM面渣 · 类加载](./JVM面渣级口述.md)

## 五、排查（社招 P0）

### 白话结构（任何场景都用）

```text
现象（超时/CPU/OOM）→ 指标与日志对齐时间
→ 工具缩小范围（jstat/jstack/dump）
→ 根因（泄漏/参数/代码）
→ 止血 + 根治 + 验证
```

| 场景 | 第一反应 |
|------|----------|
| 堆 OOM | dump → MAT 看谁占内存、引用链 |
| Full GC 频繁 | GC 后 Old 降不降？不降偏泄漏 |
| CPU 100% | top 线程 → jstack 对栈 |
| 容器被杀无 Java OOM | RSS 超 cgroup，堆外也占 limit |

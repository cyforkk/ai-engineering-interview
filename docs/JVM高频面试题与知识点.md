## 1. 内存分区：像公司租办公楼

| 地方 | 人话 | 谁用 |
|------|------|------|
| 栈 | 每个人自己的**工位抽屉**（方法调用、局部变量） | 每个线程一份 |
| 堆 | **公司大仓库**，货物（对象）放这 | 大家共享，GC 主要清这里 |
| 方法区/元空间 | **说明书柜**（类的信息） | 共享 |
| 程序计数器 | 「我执行到哪一行了」便利贴 | 每线程一份 |

### 堆为啥还分新生代、老年代？

经验：很多对象用一下就扔（外卖盒），少数活很久（工位电脑）。  

- **新生代**：快生快死，勤快打扫  
- **老年代**：活得久的，打扫方式不同  

这样比「整栋楼每次大扫除」划算。

---

## 2. 垃圾怎么认？

### 先记一句

从「还被用到的根」出发，沿引用走，**走不到的就是垃圾**。

不是简单数「有几个箭头指着我」（循环引用会数不清）。

### GC 为啥有时让程序顿一下？

打扫仓库时，有时要暂停业务员进出（停顿）。  
Full GC 像**大扫除**，通常更痛；要尽量少发生。

G1 可以想成：仓库划成很多格子，优先清**垃圾最多的格子**，好控制停顿时间。

## 3. 类加载双亲委派（一句话故事）

加载一个类时，先问「爸爸加载器会不会」，爸爸不会再自己加载。  
为啥？避免你写个假的 `java.lang.String` 把系统核心类顶掉。

## 4. 线上问题：先会这三招

### 堆内存溢出（OOM）

像仓库爆满。  
**思路：** 留现场（dump）→ 看谁占地方最多、为啥没被扔掉（有没有被错误引用吊着）。

### Full GC 很频繁

先看：打扫完老年代**有没有降下去**。  
降不下去，像垃圾清不走（泄漏）；能降又涨，像生产垃圾太快。

### CPU 100%

像员工疯转。  
**思路：** 找到最忙的线程 → 看它堆栈在干啥（死循环？狂算？锁？）。

### 容器里进程被杀，但 Java 没报 OOM

可能是整栋楼电费限额（cgroup）到了，**堆外内存**也算占用。只盯 `-Xmx` 不够。

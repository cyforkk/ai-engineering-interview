## 一、本专题在考什么

概念（内存/GC/类加载）是基础；**社招分水岭**是：线上 OOM、Full GC、CPU 飙高时，你是否有**可复述的排查路径**，以及每一步为什么那样做。

---

## 二、运行时内存：为何这样切

| 区域 | 为何独立 |
|------|----------|
| 程序计数器 | 线程切换后要知道执行到哪；线程私有最简单 |
| 虚拟机栈 | 方法调用天然栈式；私有避免干扰 |
| 堆 | 对象共享、跨方法存活；GC 主战场 |
| 元空间 | 类元数据；移出堆避免永久代难调与易 OOM |
| 直接内存 | NIO 堆外，绕过部分堆限制，但受本机与容器 limit 约束 |

### 分代假说

「大多数对象朝生夕死」→ 新生代用复制算法廉价回收；活得久的进老年代，用不同策略。这是经验假设，不是数学定理，但与大量 Java 服务匹配。

---

## 三、GC：为何可达性而不是引用计数

引用计数对**循环引用**无能为力（A↔B 计数永不为 0 但已不可达）。可达性从 GC Roots 出发标记活对象，更符合语言语义。

### 算法与分代匹配

- 新生代活对象少 → **复制**到另一半，简单  
- 老年代活对象多 → **标记-整理/清除**，避免大量复制  
- G1：堆切 Region，可设定停顿目标，优先清垃圾多的 Region——为了**可控停顿**而不是极致吞吐 alone  

### Full GC 为何可怕

停顿更长、范围更大；频繁 Full GC 常是分配过快、晋升失败、老年代碎片或泄漏导致「越收越多」。

---

## 四、双亲委派：为什么

若任意应用 ClassLoader 都能先加载 `java.lang.String`，核心类可被替换 → 安全与一致性崩溃。  
先委派父加载：核心类由启动类加载器加载，保证唯一。  
破坏双亲（SPI、热部署、容器隔离）是为了**插件/隔离**需求，属于有意识的例外。

---

## 五、排查：为何这套顺序

```text
现象对齐时间 → 指标（GC/CPU/内存）→ 工具缩小范围 → 根因 → 止血与根治
```

- **先对齐时间**：避免把无关日志当原因  
- **GC 后 Old 降不降**：不降更像泄漏；降了又涨像分配过快  
- **dump 要在问题复现时**：事后空跑 dump 可能看不到元凶  
- **容器 limit**：只看 `-Xmx` 不够，RSS 含堆外与其他

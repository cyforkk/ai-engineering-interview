# 一、超高频（几乎必问）

## 1. 运行时数据区（内存结构）

### 1.1 有哪些区域？私有还是共享？

| 区域 | 线程 | 存储内容 |
|------|------|----------|
| **程序计数器** | 私有 | 当前字节码指令地址；切换线程后恢复执行位置 |
| **虚拟机栈** | 私有 | 栈帧：局部变量表、操作数栈、动态链接、方法出口 |
| **本地方法栈** | 私有 | native 方法相关 |
| **堆** | 共享 | 对象实例、数组；GC 主战场 |
| **方法区** | 共享 | 类元数据、运行时常量池、静态变量等（实现见元空间） |
| **直接内存** | 堆外 | 非运行时数据区严格一部分，但受本机限制；NIO 常用 |

- 规范：程序计数器是唯一不会规定 OOM 的区域（一般不抛 OOM）。  
- 栈：`StackOverflowError`；堆/方法区/直接内存：多种 `OutOfMemoryError`。

### 1.2 堆的分代结构？

```text
堆 Heap
├── 新生代 Young（通常较小）
│    ├── Eden
│    ├── Survivor From (S0)
│    └── Survivor To   (S1)
└── 老年代 Old / Tenured
```

- **分代假说**：多数对象朝生夕死 → 新生代用复制算法高频回收。  
- 长期存活、大对象等进老年代。  
- G1 逻辑上仍分年轻/老，物理上是 **Region** 拼起来。

### 1.3 方法区 / 永久代 / 元空间演变？

| JDK | 实现 | 位置 |
|-----|------|------|
| ≤6 | 永久代 PermGen | 堆的一部分（逻辑） |
| 7 | 逐步迁移（如字符串常量池进堆） | 过渡 |
| **8+** | **元空间 Metaspace** | **本地内存** |

- **为什么弃永久代、改元空间？**  
  - 永久代大小难调，易 `PermGen OOM`  
  - 元空间用本地内存，受机器内存限制，更灵活  
  - 类元数据与 Java 堆解耦  

### 1.4 直接内存是什么？场景？

- 堆外内存，通过 NIO `ByteBuffer.allocateDirect` 等分配。  
- **场景**：高频网络 IO、减少用户态/内核态拷贝路径、堆压力大时卸载缓冲。  
- **注意**：也受 `-XX:MaxDirectMemorySize` 与容器 limit；泄漏难查，需堆外监控。

### 1.5 栈溢出 vs 堆溢出？

| | StackOverflowError | OutOfMemoryError: Java heap space |
|--|--------------------|-----------------------------------|
| 典型 | 递归过深、栈帧过大 | 堆不足、泄漏、分配过猛 |
| 区 | 虚拟机栈/本地方法栈 | 堆 |
| 其它 OOM | 栈扩展失败也可能 OOM（实现相关） | Metaspace、Direct buffer 等见下文 |

## 2. 垃圾回收核心

### 2.1 如何判断对象存活？

| 方法 | 原理 | 问题 |
|------|------|------|
| 引用计数 | 引用数 0 可回收 | **循环引用**无法回收 |
| **可达性分析** | 从 GC Roots 出发不可达则可回收 | Java 主方案 |

### 2.2 什么是 GC Roots？常见有哪些？

- 可达性分析的**起点集合**。  
- 常见：  
  - 虚拟机栈中引用的对象  
  - 方法区静态属性引用的对象  
  - 方法区常量引用的对象  
  - 本地方法栈 JNI 引用  
  - 同步锁持有的对象  
  - JVM 内部引用（系统类加载器、异常对象等，了解）  

### 2.3 四种垃圾收集算法？

| 算法 | 原理 | 优点 | 缺点 | 典型 |
|------|------|------|------|------|
| 标记-清除 | 标垃圾再清 | 实现简单 | **碎片** | CMS 老年代类 |
| 复制 | 活对象拷到另一半 | 无碎片、快 | 空间浪费 | 新生代 |
| 标记-整理 | 标后滑动整理 | 无碎片 | 移动成本 | 老年代 |
| 分代收集 | 不同代不同算法 | 综合效率 | 实现复杂 | 现代基础 |

### 2.4 Minor / Major / Full GC？

| 名称 | 常见含义 |
|------|----------|
| **Minor GC / Young GC** | 主要回收新生代 |
| **Major GC** | 常指老年代（说法不统一，面试说明自己定义） |
| **Full GC** | 整堆或含方法区/元空间等大范围回收，**STW 通常更长** |

- 触发直觉：Eden 满 → Young GC；老年代不足、元空间不足、显式 `System.gc()`、晋升失败等 → 易 Full（与收集器有关）。  

### 2.5 对象何时进老年代？

1. **年龄阈值**：Survivor 熬过多次 GC，年龄到阈值（默认常 15，可调）  
2. **大对象**：直接进老年代（如 `-XX:PretenureSizeThreshold`，Serial 等；G1 策略不同）  
3. **动态年龄**：Survivor 中相同年龄对象大小总和超过 Survivor 一半等规则，可提前晋级  
4. **空间分配担保**：Minor 前预判老年代是否放得下存活对象，不够则可能 Full  

### 2.6 STW 是什么？为什么需要？

- **Stop-The-World**：GC 时暂停用户线程。  
- **为什么**：保证一致性视图（对象图在标记时不能乱变）。  
- 低延迟收集器目标是 **缩短/减少 STW**，而非完全没有。

## 3. 垃圾收集器

### 3.1 常见收集器一览

| 收集器 | 代 | 特点 |
|--------|-----|------|
| Serial | 新/老 | 单线程，Client 简单场景 |
| ParNew | 新 | Serial 并行版，常配 CMS |
| Parallel Scavenge / Old | 新/老 | **吞吐优先** |
| **CMS** | 老 | 并发标记清除，低停顿；**已废弃趋势** |
| **G1** | 整堆 Region | 可预测停顿，JDK9+ 常见默认 |
| **ZGC** | 整堆 | **超低延迟**（毫秒级目标） |
| Shenandoah | 整堆 | 低延迟（了解） |

### 3.2 CMS 流程与优缺点？为何废弃？

**流程（主干）：**

```text
初始标记(STW) → 并发标记 → 重新标记(STW) → 并发清除
```

| 优点 | 缺点 |
|------|------|
| 并发，停顿相对小 | **碎片**（标记-清除） |
| | 浮动垃圾 |
| | 对 CPU 敏感 |
| | 并发失败 → 退化 Full |

- **废弃原因**：G1/ZGC 更好；碎片与维护成本；JDK9+ 弃用，14 移除。  

### 3.3 G1 核心原理？

- 堆拆成等大 **Region**（可属于 Eden/Survivor/Old/Humongous）。  
- **优先回收垃圾占比高**的 Region（Garbage First）。  
- **Remembered Set (RSet)**：记录跨 Region 引用，避免全堆扫。  
- **可预测停顿**：设定停顿目标，按收益选回收集合（CSet）。  
- 年轻代回收 + **Mixed GC**（年轻 + 部分老 Region）。  

### 3.4 CMS vs G1？

| | CMS | G1 |
|--|-----|-----|
| 内存布局 | 连续分代 | Region |
| 算法 | 标记-清除（碎片） | 整理为主，减少碎片 |
| 停顿 | 低但不可控碎片 Full | 更追求可控停顿 |
| 适用 | 旧系统 | 大堆、要控停顿的主流 |

### 3.5 ZGC / Shenandoah？

- **目标**：超低延迟（大堆下停顿仍短）。  
- ZGC：**染色指针**、读屏障、并发整理（面试点到关键词）。  
- 加分项：知道适用超大堆、低延迟业务。

## 4. 类加载机制

### 4.1 完整过程？

```text
加载 → 验证 → 准备 → 解析 → 初始化
（链接 = 验证 + 准备 + 解析）
```

| 阶段 | 做什么 |
|------|--------|
| 加载 | 字节流 → 方法区类结构，生成 Class 对象 |
| 验证 | 格式、元数据、字节码、符号引用 |
| 准备 | 静态变量分配内存并设 **零值** |
| 解析 | 符号引用 → 直接引用 |
| 初始化 | 执行 `<clinit>`（静态赋值与静态块） |

### 4.2 双亲委派？流程？

```text
类加载请求
 → 先委派父加载器
 → 父一路向上
 → 父不能加载再自己加载
```

### 4.3 为什么需要双亲委派？

1. **安全**：防止自定义 `java.lang.String` 等核心类被替换  
2. **避免重复加载**：保证类的唯一性（同一加载器命名空间）  

### 4.4 如何打破？

| 场景 | 做法直觉 |
|------|----------|
| **SPI**（JDBC） | 线程上下文类加载器加载实现 |
| **Tomcat** | WebApp 优先加载自己的类，隔离应用 |
| 热部署 | 自定义 ClassLoader 丢弃重建 |
| OSGi | 网状委托 |

### 4.5 常见类加载器？

| 加载器 | 加载范围 |
|--------|----------|
| Bootstrap 启动 | `$JAVA_HOME/lib` 核心（rt 等，实现相关） |
| Extension / Platform | 扩展目录 |
| Application 应用 | classpath |
| 自定义 | 继承 ClassLoader |

## 5. 对象相关

### 5.1 创建过程（五步）

1. **类加载检查**  
2. **分配内存**（指针碰撞 / 空闲列表；可配合 **TLAB**）  
3. **初始化零值**  
4. **设置对象头**  
5. **执行 `<init>` 构造**  

### 5.2 内存布局？

```text
┌─────────────────────────────┐
│ 对象头 Object Header        │
│  · Mark Word（hash/锁/年龄）│
│  · Klass Pointer（类型指针）│
│  · 数组长度（仅数组）        │
├─────────────────────────────┤
│ 实例数据 Instance Data      │
├─────────────────────────────┤
│ 对齐填充 Padding（8 字节）   │
└─────────────────────────────┘
```

- 与 **synchronized 锁升级**（改 Mark Word）强相关。  

### 5.3 访问定位：句柄 vs 直接指针？

| 方式 | 优点 | 缺点 |
|------|------|------|
| 句柄 | 对象移动只改句柄 | 多一次间接 |
| **直接指针** | 访问快 | 移动对象要改引用 |

- **HotSpot 主要用直接指针**。

# 二、高频

## 6. 引用类型

| 类型 | 回收时机 | 场景 |
|------|----------|------|
| 强 | 不达则不收 | 默认 |
| 软 | 内存紧张易收 | 图片/对象缓存 |
| 弱 | 下次 GC 可能就收 | ThreadLocalMap key、WeakHashMap |
| 虚 | 形同无，用于跟踪回收 | 堆外释放管理等 |

## 7. 内存问题排查

### 7.1 常见 OOM 类型？

| 类型 | 方向 |
|------|------|
| Java heap space | 堆泄漏/不足 |
| Metaspace | 类太多/动态生成类泄漏 |
| Direct buffer memory | 堆外 |
| GC overhead limit exceeded | GC 太频繁几乎不回收有效内存 |
| Unable to create native thread | 线程过多 |

### 7.2 泄漏 vs 溢出？

| | 泄漏 Leak | 溢出 OOM |
|--|-----------|----------|
| 含义 | 用完未释放，可用内存渐少 | 申请失败 |
| 关系 | 泄漏常导致最终溢出 | 也可能配置过小瞬时溢出 |

### 7.3 如何排查？

```text
复现/观察 → GC 日志与监控
 → jmap dump / -XX:+HeapDumpOnOutOfMemoryError
 → MAT 看 Histogram、Dominator Tree、引用链
 → 结合代码修泄漏或调参
```

- 工具：jmap、jstat、jstack、MAT、Arthas、VisualVM。  
- **CPU 飙高**：`top -H` 找线程 → jstack 对栈。

## 8. JVM 参数与调优基础

### 8.1 常用参数

| 参数 | 含义 |
|------|------|
| -Xms / -Xmx | 堆初始/最大（常相同） |
| -Xmn | 年轻代大小（G1 场景慎用传统调法） |
| -XX:SurvivorRatio | Eden:Survivor 比例 |
| -XX:MetaspaceSize / MaxMetaspaceSize | 元空间 |
| -XX:+UseG1GC | G1 |
| -XX:MaxGCPauseMillis | G1 停顿目标 |
| -XX:+HeapDumpOnOutOfMemoryError | OOM dump |
| -Xlog:gc*（JDK9+）/ -XX:+PrintGCDetails | GC 日志 |

### 8.2 如何设堆？代比例？

- 堆：根据容器 limit 与峰值，留 OS/堆外余量；**压测**定。  
- 传统分代：年轻代过小 → Minor 频繁；过大 → Full 间隔但单次晋升压力等，需观测。  
- G1：更关注停顿目标与 Region，而非死磕 -Xmn。  

### 8.3 Full GC 何时发生？如何减少？

- 老年代不足、元空间不足、晋升失败、显式 GC、大对象等。  
- 减少：修泄漏、合理堆与代、避免无必要 System.gc、优化分配速率、换 G1/ZGC、检查元空间与堆外。

## 9. 其他高频点

### 9.1 字符串常量池在哪？

- **JDK7+ 在堆中**（从永久代移出）。  
- `intern()` 行为与版本相关，7+ 池在堆。  

### 9.2 TLAB？

- Thread Local Allocation Buffer：线程私有分配缓冲，减少堆分配时的同步竞争。  

### 9.3 逃逸分析？

- 分析对象是否逃逸出方法/线程。  
- 优化：**栈上分配**、**标量替换**、**锁消除**（JIT 配合）。  
- 不一定真正「栈上分配对象」，常以标量替换等形式体现。

# 三、中频

## 10. JIT 与解释执行？

- 解释器快速启动执行字节码。  
- **JIT** 对热点代码编译成本地机器码加速。  
- 分层编译（C1/C2）平衡启动与峰值性能。  

### 方法内联

- 小方法体嵌入调用处，减调用开销，便于其它优化。  

## 11. 工具清单

| 工具 | 用途 |
|------|------|
| jps | 进程 |
| jstat | GC/类统计 |
| jmap | 堆 dump、histo |
| jstack | 线程栈 |
| jinfo | 参数 |
| JConsole / VisualVM | 图形监控 |
| Arthas | 在线诊断 |
| MAT | dump 分析 |

### GC 日志

- 看 Young/Full 频率、停顿时间、堆使用前后变化。  
- JDK9+ 统一日志：`-Xlog:gc*`。  

### CPU 飙高

```text
top 找进程/线程 → printf %x 转 nid → jstack 对栈
```

### 元空间溢出常见原因

- 大量动态代理/增强、热部署类加载器未卸载、反射生成类过多。  

### 直接内存溢出排查

- 限制 MaxDirectMemorySize；查 NIO/Netty 是否未释放；Native Memory Tracking（进阶）。
